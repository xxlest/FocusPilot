## 二、逐产品开发者能力详解

### 1. Rokid Glasses（国产首选）

**系统**：YodaOS-Sprite，AOSP 分支，2025-09 首次开放 SDK。

**开发模型**：

| 路线 | 成熟度 | 说明 |
|---|---|---|
| **CXR-M SDK（手机端）** | ⭐⭐⭐ 官方主推 | Kotlin/Java，Maven 包 `CxrApi`。手机 App 通过 BLE 连眼镜，拉音频/推送显示内容 |
| **眼镜端原生 APK** | ⭐ 可行但零案例 | ADB 侧载成功案例有，但官方文档稀疏 |
| **UXR SDK (Unity)** | ❌ 已废弃 | 针对老款 AR Studio/Lite，2021 归档，Glasses 不能用 |

**开发者可用的能力**：

| 能力 | 开放状态 | API/方式 |
|---|---|---|
| 麦克风原始 PCM | ✅ | `CxrApi.openAudioRecord(codecType, streamType)` + `AudioStreamListener`，16kHz/16bit/mono |
| 内置 ASR | ❌ 无 | 必须在手机端自己接（讯飞/Whisper/百度） |
| 自定义唤醒词 | ❌ | 「Hi Rokid」锁死 |
| HUD 文本显示 | ✅ | `sendAsrContent()` / `sendTtsContent()` |
| HUD 自定义图形 | ❌ | 无绘制 API，仅文本推送 |
| 触控条事件 | ❌ | 无公开 API |
| WiFi 网络 | ✅ 眼镜有 | 手机端 CXR-M 路线走手机网络；眼镜端 APK 理论可用 |
| 后台常驻 | ⚠️ | 无明确文档，社区反馈掉电快 |

**申请门槛**：报名「开发者尝鲜计划」→ 提交策划案 → 完成 Roki Academy 课程 → 获发磁吸调试线（ADB 才能用）

**关键文档**：
- 开放平台：https://ar.rokid.com/sdk
- 开发者文档：https://developerdoc.rokid.com/
- 开发者论坛：https://forum.rokid.com/
- Maven：`https://maven.rokid.com/repository/maven-public/`
- 开源示例：[RokidAIAssistant](https://github.com/zero2005x/RokidAIAssistant)

**FocusPilot 对接方案**：手机端写 Kotlin App → `CxrApi.openAudioRecord()` 拿 PCM → Whisper ASR → HTTP POST 到 Mac。**需要一台 Android 手机常挂做中继**。

**典型二开场景**：
- 实时字幕翻译（会议/旅行场景，拉 PCM → 云 ASR+翻译 → 推回 HUD 显示）
- 仓储/物流拣货指引（扫码识别 → HUD 显示货架路径）
- 钉钉会议 HUD 提词器（手机端拉钉钉消息 → 推送到眼镜 HUD）

**厂商心态**：Rokid 的核心商业模式是 **B 端行业解决方案**（仓储、安防、远程协作），消费版 Glasses 是品牌入口和开发者生态培育工具。开放 CXR-M SDK 的目的是**让集成商和行业开发者用手机做"大脑"、眼镜做"眼和耳"**，降低行业客户的定制门槛。消费者二开只是顺便受益。这也解释了为什么眼镜端 APK 路线文档稀疏——Rokid 并不鼓励开发者绕开手机端直接在眼镜上折腾。

### 2. Brilliant Labs Frame（极客/开源首选）

**系统**：nRF52840 芯片上的 Lua VM 运行时（裸机，非 RTOS）。

**开发模型**：Host App 模式——Python/Flutter SDK 在宿主设备上运行，通过 BLE 与 Frame 通信，把 Lua 字符串 push 到眼镜执行。

**开发者可用的能力**：

| 能力 | 开放状态 | API/方式 |
|---|---|---|
| 麦克风原始 PCM | ✅ | BLE 流回宿主，4-16bit, 4-20kHz |
| 眼镜端脚本 | ✅ | 推 Lua 字符串到 Frame 执行 |
| 显示 | ✅ | 单眼 micro-OLED，低分辨率文本 |
| 摄像头 | ✅ | 拍照可回传 |
| WiFi | ❌ | 仅 BLE 5.3 |
| TTS | ✅ | 2025 更新片上 TTS |

**无门槛**：硬件软件全开源，买了就能开发，无需申请。

**关键文档**：
- 硬件手册：https://docs.brilliant.xyz/frame/hardware/
- Lua API：https://docs.brilliant.xyz/frame/frame-sdk-lua/
- Python SDK：https://docs.brilliant.xyz/frame/frame-sdk-python/

**FocusPilot 对接方案**：Mac 直接做 BLE 宿主（Python 进程）→ 拿 PCM → Whisper.cpp 本地 ASR → POST 到 localhost。**无需手机，Mac 即宿主**。

**典型二开场景**：
- 个人 AI 助手（拍照 → 视觉问答 → TTS 回读；类似 "看到什么就问什么"）
- 实时翻译/字幕（PCM 流 → Whisper → 翻译 → 回显到 micro-OLED）
- 开发者自制 HUD 工具（天气/日程/导航提示，Lua 脚本控制显示内容）

**厂商心态**：Brilliant Labs 是一家**以开源硬件为信仰的小团队**，创始人来自开源社区（Monocle 前身也是开源项目）。他们的商业模式是**卖硬件**，不靠软件生态抽成。全面开源（硬件原理图 + 固件 + SDK）的目的是**吸引极客和独立开发者成为早期用户和布道者**，靠社区口碑驱动硬件销量。Frame 不试图做平台，只试图做"开发者最爱的硬件载体"。这也解释了为什么它功能极简（无 WiFi、弱显示）——把复杂度留给开发者，自己只做好 I/O。

### 3. Ray-Ban Meta Smart Glasses Gen 2

**系统**：闭源 RTOS（基于高通 AR1），不对外。

**开发模型**：2025-12 推出 **Wearables Device Access Toolkit**（Public Developer Preview）。但官方 FAQ 明确：**"Apps don't run on the glasses themselves"**。开发者写 iOS/Android 手机 App，通过 Toolkit 访问眼镜的 mic/camera 数据流。

**开发者可用的能力**：

| 能力 | 开放状态 | API/方式 |
|---|---|---|
| 麦克风/摄像头 | ✅ 预览中 | Toolkit 提供 sensor access，具体格式待全量发布 |
| 眼镜端代码 | ❌ | 不可能 |
| 自定义语音助手 | ❌ | Meta AI 不可替换 |
| Mac SDK | ❌ | 仅 iOS/Android |
| WhatsApp 语音发送 | ✅ 原生 | "Hey Meta, send WhatsApp to X"（仅英文，大陆不可用） |

**门槛**：目前仅限选定合作伙伴，普通开发者只能本机原型测试，2026 预计全量开放。

**关键文档**：
- Toolkit FAQ：https://developers.meta.com/wearables/faq/
- 博客公告：https://developers.meta.com/blog/introducing-meta-wearables-device-access-toolkit/

**FocusPilot 对接**：❌ 无 macOS SDK，Toolkit 强制手机做宿主，且大陆不可激活。**不适合**。

**典型二开场景**：
- 视觉无障碍辅助（摄像头拍路况 → AI 识别 → 语音播报，Be My Eyes 已接入）
- 实时翻译（Meta AI 内置，但第三方可通过 Toolkit 做更专业的行业翻译）
- 社交媒体直播/第一人称拍摄（Instagram/Facebook Live 已原生支持）

**厂商心态**：Meta 做眼镜的核心战略是**用 Ray-Ban 品牌力做消费者触达，构建 Meta AI 的可穿戴入口**。开放 Toolkit 的目的不是让开发者自由创造，而是**筛选高质量合作伙伴来丰富 Meta AI 的能力边界**（所以初期限定合作伙伴，不开放大众注册）。Meta 不希望第三方替换 Meta AI，只希望第三方"喂数据给 Meta AI"。这是典型的**平台型封闭生态**：开放恰到好处的 I/O，但控制大脑。对独立开发者来说，你不是在"做自己的 App"，而是在"给 Meta 的生态打工"。

### 4. Even Realities G1

**系统**：闭源 RTOS，左右镜腿各一颗 MCU + 各一路独立 BLE。

**开发模型**：当前固件级功能集合，**2026 即将推出 Even Hub 第三方 App 平台**（HUD 轻量 App）。社区已有基于逆向 BLE 协议的 [EvenDemoApp](https://github.com/even-realities/EvenDemoApp)。也可走 MentraOS 平台。

**开发者可用的能力**：

| 能力 | 开放状态 | API/方式 |
|---|---|---|
| 麦克风 | ✅ | 实时音频流，**LC3 编码**（非 PCM，需解码） |
| 显示 | ✅ | 双眼绿色 HUD，通过 BLE 协议推内容 |
| 通知显示 | ✅ | 可白名单 IM App 通知 |
| 回复消息 | ❌ | 只能"看"不能"回" |
| WiFi | ❌ | 仅 BLE |
| 第三方 App | ⚠️ | Even Hub 即将开放 |

**门槛**：当前需逆向 BLE 协议或走 MentraOS。

**FocusPilot 对接方案**：Mac BLE 直连，拿 LC3 音频流 → 解码 → ASR → HTTP。**可行但需处理 LC3 解码**。

**典型二开场景**：
- 演讲/会议提词器（BLE 推文本到双眼 HUD，演讲者无感看稿）
- 实时字幕（MentraOS 已有 LiveCaptions App）
- 笔记速记（TouchBar 长按录音 → QuickNote，但只读不回）

**厂商心态**：Even Realities 是一家**以"隐形感"为核心卖点的硬件公司**（G1 外观几乎和普通眼镜无异，39g）。早期策略是**先靠极致轻量+好看的外观打消费者市场**，开发者生态是后补。2026 推出 Even Hub 说明它开始意识到：光靠自研固件功能（翻译/提词/导航）增长有限，**需要第三方 App 来扩充使用场景、提升用户粘性**。但开放得很谨慎——Hub 只做 HUD 轻量 App，不打算让开发者深度控制硬件。本质上是"先卖好硬件，再慢慢开放软件"。

### 5. 小米 AI 眼镜

**系统**：高通骁龙 AR1（推测基于 AOSP，中科创达深度定制）+ 恒玄 BES2700（RTOS 常驻语音/低功耗）。官方未公开 OS 名称。

**开发模型**：**完全封闭**。截至 2026Q1：
- 零开发者 SDK
- 零第三方 App 通道
- 零 sideload 能力
- 麦克风/网络对第三方不开放
- 生态由蜂巢科技等指定厂商预装接入

**门槛**：无门可入。

**评测口碑**：36氪/南都/知乎多篇评测一致反馈 AI 功能"几乎全线翻车"，生态联动不稳定。

**FocusPilot 对接**：❌ **完全不可行**。

**典型二开场景**：无。当前唯一的"开发"是蜂巢科技等小米指定合作伙伴的预装 App（视频通话/直播）。

**厂商心态**：小米做眼镜的战略是**AIoT 生态的延伸，不是开发者平台**。小米的思路一贯是"硬件走量 + 自有服务闭环"——超级小爱是入口，小米生态链产品（手环/音箱/家电）是触达面，眼镜只是新增的一个"小爱终端"。不开放 SDK 的原因很直接：**小米不需要第三方开发者来定义眼镜的功能**，它需要的是让眼镜融入自己的 AIoT 生态（控制家电、拍照翻译、导航），而这些功能小米自己做就够了。开放 SDK 意味着质量失控、安全风险、品牌稀释——对小米这种走量品牌来说得不偿失。**除非销量遇到瓶颈倒逼生态开放，否则短期不会有 SDK**。

### 6. Vuzix Z100 / Blade 2

**Z100**：闭源 RTOS，peripheral 模式（自身不跑 App），靠手机 Vuzix Connect App 推送控制。开发者走 Android/iOS SDK 写手机端 App。2025 开始支持 AugmentOS 第三方 App 平台。

**Blade 2**：**Android 11**（非初代 Blade 的 5.1），标准 Android Studio 装 APK，有镜腿触控板。理论上能直接装 APK + WiFi 发 HTTP，但已停产，偏工业场景，$1000+ 价位。

**FocusPilot 对接**：Blade 2 理论直推可行但不实际（停产+工业向）；Z100 需手机中转。

**典型二开场景**：
- 仓储拣货（Z100：扫码 → HUD 显示货架位置 → 语音确认，DHL/UPS 已部署）
- 远程专家协作（Blade 2：现场技术员第一人称视频 → 远端专家标注 → HUD 叠加指引）
- 医疗查房辅助（Z100：患者信息 HUD 显示，解放双手）

**厂商心态**：Vuzix 是**纯 B 端工业眼镜公司**，上市企业，客户是 DHL、三星、通用汽车。开放 Android SDK（Blade 2）和 AugmentOS 平台（Z100）的目的是**让行业 SI 集成商快速做定制方案**。消费市场不是 Vuzix 的菜——Z100 的 38g 和 2 天续航是为"8 小时工厂轮班"优化的，不是为你戴着逛街。这也是为什么 Blade 2 停产了：工业客户更需要轻便长续航的 Z100，不需要全功能 Android。

### 7. XREAL One / Air 系列

**本质**：纯 OLED 外接显示器，眼镜本体无通用 OS、无计算单元。X1 协处理器只跑 3DoF 低延迟渲染。

**NRSDK** 跑在 Android 手机/Beam Pro 上，眼镜只接 DP-Alt 视频信号。Mac 上可识别为 USB 外接显示器+USB 音频设备，但 NRSDK 不支持 macOS。

**FocusPilot 对接**：❌ 场景不匹配，这是显示设备不是输入设备。

**典型二开场景**：
- 大屏办公/多屏扩展（Mac 连 XREAL Air 当第二块显示器）
- 游戏/影视沉浸体验（Steam Deck / PS5 接 XREAL One）
- AR 空间应用（Unity NRSDK，但仅限 Android 宿主）

**厂商心态**：XREAL 的定位很清晰——**做"空间显示器"品类的领导者**，不做输入、不做 AI、不做平台。NRSDK 开放给开发者是为了**让游戏/应用适配 XREAL 的显示效果**（3DoF 头追、空间锚定），本质是"帮我做内容，让用户觉得买 XREAL 值"。XREAL 对"眼镜当输入设备"完全不感兴趣——它的商业逻辑是卖显示硬件，内容由宿主设备提供。

---

### 8. Solos AirGo V2

**系统**：闭源固件。

**开发模型**：**付费 SDK Developer Program**（$1999/年，含两副眼镜+技术支持）。iOS/Android 双端 SDK，BLE 控制 + WiFi 数据通道。暴露 webhook hook。

**FocusPilot 对接**：必须经手机 SDK 中转，且门槛高（$1999/年）。**企业可选，个人不推荐**。

**典型二开场景**：
- 运动数据 HUD（骑行/跑步时心率、配速、导航叠加显示）
- 语音笔记（骑行/开车时说一句话，App 记录并同步云端）
- 健康监测（与智能手表联动，HUD 显示提醒）

**厂商心态**：Solos 定位是**运动+健康领域的垂直智能眼镜**（创始人来自运动光学行业）。$1999/年 SDK 付费门槛说明它**不追求开发者数量，追求的是高质量行业合作伙伴**（健身品牌/运动 App/企业客户）。开放 SDK 是为了让 Strava、Garmin 这类运动 App 能做定制集成，不是为了让独立开发者自由发挥。对小团队来说这个门槛是刻意设置的筛选器。

---

### 9. MentraOS / AugmentOS

**不是设备 OS，是跨设备 SDK 中间件平台**（MIT 开源）。

**架构**：眼镜 → 手机 BLE → MentraOS 云 → 你的 App Server（TypeScript SDK）。支持 Even G1、Vuzix Z100 等多款硬件。

**开发者能力**：TypeScript 写 App 跑在你的服务器上，通过 MentraOS Cloud 收 mic/camera 事件、回写 display。

**FocusPilot 对接**：可行但绕远路（必经手机+云）。适合想一套代码兼容多款眼镜的场景。

**典型二开场景**：
- 跨设备统一字幕/翻译（一套 TS 代码同时跑在 G1、Z100、Frame 上）
- 第三方 AI 助手（替换厂商自带助手，做自定义语音交互）
- 企业统一管理平台（一个后台管多款眼镜设备的部署和内容推送）

**厂商心态**：MentraOS 的野心是做**智能眼镜界的 Android**——一个跨硬件的统一操作系统/中间件层。MIT 开源是为了**快速圈地**，趁各家眼镜厂商的自有生态还没成熟，先用开源社区的力量建立事实标准。它的商业模式大概率是"开源免费 + 云服务收费 + 企业版付费支持"。对开发者来说 MentraOS 最大的价值是**"写一次跑多个眼镜"**，但代价是必须接受"手机+云"的中转架构（因为 MentraOS 需要控制中间层来保持跨设备兼容性）。

**文档**：
- GitHub：https://github.com/Mentra-Community/MentraOS
- Docs：https://docs.mentraglass.com

## 四、FocusPilot 对接结论

### 核心发现

**智能眼镜当前的开发范式不是"在眼镜上跑 App"，而是"眼镜做 I/O、外部设备做大脑"**。即使是号称 AOSP 的 Rokid，官方主推的也是手机端 SDK。

这意味着对 FocusPilot 来说，对接的问题其实很简单：

```
眼镜（麦克风采音）
  └─ BLE / WiFi ─→ 宿主设备（手机 or Mac）
                     ├─ ASR 转写
                     └─ HTTP POST → FocusPilot InputBridgeService
```

**FocusPilot 只需要做一件事：提供一个 HTTP 接收接口。** 硬件适配全部在外围解决。

> 补充说明：这个判断适合作为总体方向，但在主方案中已经进一步收敛为“FocusPilot 需要的是统一语音输入桥，而不只是一个 HTTP 接收接口”。HTTP 只是接入形式之一，最终还需要事件模型、意图路由、反馈闭环和安全落点。

### 推荐路径

| 优先级 | 方案 | 硬件 | 链路 | 工程量 |
|---|---|---|---|---|
| P0 | iPhone Siri + Shortcuts | 任意蓝牙耳机/眼镜当麦克风 | Siri 听写 → Shortcut → HTTP POST → Mac | **一晚上** |
| P1 | Brilliant Frame + Mac BLE | Frame $349 | Frame → BLE → Mac Python → 本地 ASR → HTTP | 中（Python 守护+Whisper） |
| P2 | Rokid Glasses + Android App | Rokid ¥2499 + Android 手机 | 眼镜 → BLE → 手机 CXR-M → ASR → HTTP → Mac | 中（Kotlin App+申请开发者） |

> 补充说明：以上优先级仍然成立，但完整方案已经把标准明确为“验证成本 / 用户价值 / 维护复杂度”，因此真正的主线是：P0 先验证语音工作流价值，P1 再做 Brilliant Frame 原型，P2 才做 Rokid 消费形态试点。

### FocusPilot 侧唯一需要的改动

新增 `Services/InputBridgeService.swift`：
- 监听 `127.0.0.1:7878` HTTP POST `/inject`
- Body: `{"target": "ai_tab" | "kanban_search", "text": "..."}`
- 通过 NotificationCenter 转发到对应 View
- 这个接口与硬件无关，做好后 P0/P1/P2 三条路线都能接

> 补充说明：这部分在主方案中已经扩展。除了 `InputBridgeService`，还建议增加 `VoiceIntentRouter`、`VoiceActionExecutor`、`VoiceFeedbackService` 和默认安全落点 `Inbox`，否则只能做到“收文本”，还做不到“收工作流意图”。

## 五、与主方案的整合结论

基于这份调研，结合 FocusPilot 当前架构，最终推荐方案如下：

1. 先把 FocusPilot 定义成“可接入可穿戴语音输入的工作流中枢”，而不是“先适配某一副眼镜”。
2. 先做统一输入桥和产品闭环，再做具体硬件接入。
3. 硬件选型顺序按验证价值排序，而不是按开放度排序：
   - P0：系统语音 / iPhone Shortcut / 蓝牙耳机
   - P1：Brilliant Frame
   - P2：Rokid
4. 本调研报告继续作为硬件证据库保留，主决策和实施方案统一以 [FocusPilot-智能眼镜完整方案.md](/Users/bruce/Workspace/2-Code/01-work/FocusPilot/docs/专题研究/智能眼镜/FocusPilot-智能眼镜完整方案.md) 为准。
