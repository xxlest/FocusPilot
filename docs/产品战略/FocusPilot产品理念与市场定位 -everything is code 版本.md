# FocusPilot · 个人自迭代 AIOS 产品定位

> **FocusPilot 核心定位：个人自迭代 AIOS。**
> **采集信息，沉淀知识，实践智慧，内化能力。FocusPilot 驱动个人持续自迭代。**
>
> - **Everything is Code. Projects ship. Systems compound. FocusPilot runs yours.**
> - **Code 构建能力，Project 交付价值，FocusPilot 驱动你的个人系统持续自迭代。**

## 一、核心定位

**FocusPilot 不是任务管理工具，而是面向个人的自迭代 AIOS。**

它解决的核心问题不是“如何记住更多任务”，而是“如何让有限注意力持续投入到认知升级和方向判断中，同时保持高密度执行”。

在 FocusPilot 中，Project 用来组织认知，Task 用来表达可执行意图，Markdown 用来承载长期记忆，Agent 用来完成具体执行，反馈用来重构下一轮判断。人不再是每个任务的亲自执行者，而是整个自迭代系统的设计者、调度者和验收者。

**认知提升 × 密集做事 × 反馈重构 = 自我迭代。FocusPilot 就是让这条公式自动运转的个人 AIOS。**

**FP 真正依赖的不是某个笔记应用，而是一份开放的 Markdown + frontmatter 契约。**

如果用户没有现成笔记系统，FP 应该能自己创建和管理一个 Markdown workspace；如果用户已有 Obsidian / Logseq / 纯文件夹 / 自建工具，FP 则通过同一份 Markdown 契约接入。换句话说，笔记系统是可替换外壳，Markdown 工作记忆才是核心资产。

更短的对外表达：

> **FocusPilot 是个人自迭代 AIOS：用 Project 聚焦认知，用 Markdown 保存记忆，用 Agent 执行任务，用反馈持续重构自我。**

---

## 二、为什么是 AIOS

FocusPilot 的抽象层级不是“任务 App”，而是“个人 AI 工作操作系统”。它具备操作系统的关键结构：记忆介质、任务调度、执行进程、权限确认、系统日志、工作入口和反馈循环。

| AIOS 抽象 | FocusPilot 对应                                            |
| --------- | ---------------------------------------------------------- |
| 文件系统  | Markdown workspace                                         |
| 进程      | Agent 执行会话                                             |
| 调度器    | Task 状态流转与执行队列                                    |
| Shell     | Dashboard / Quick Panel / 对话面板                         |
| 应用      | Claude Code / Codex / Gemini / MCP 工具                    |
| 权限系统  | 人工确认、验收、回滚                                       |
| 系统日志  | Execution Log / Reports                                    |
| 长期记忆  | Project notes / Markdown frontmatter / Knowledge artifacts |
| 桌面环境  | 今日聚焦、窗口管理、番茄钟、专注入口                       |

传统任务工具默认“人是执行主体”，所以它们解决的是记忆、提醒和排序。FocusPilot 默认“AI 可以成为执行主体”，所以它解决的是调度、验收、沉淀和自我迭代。

## 三、核心命题：专注持续自我迭代

自我迭代不是单纯学习，也不是单纯做事，而是两者形成螺旋：

- **认知提升**：通过阅读、研究、规划和复盘形成更高质量的判断；
- **密集做事**：通过高频执行把判断投入现实，获得真实反馈；
- **反馈重构**：把执行结果写回项目记忆，修正下一轮认知和行动。

FocusPilot 的角色，是把人的注意力从低价值执行中释放出来，让人持续留在更高杠杆的位置：

- 人负责专注、方向、拆解、授权和验收；
- AI 负责执行、生成、整理、记录和回写；
- Markdown 负责保存可迁移、可追踪、可被 AI 读取的长期记忆；
- Project 负责让所有行动进入闭环，而不是散落成临时任务。

## 四、核心理念（四个命题）

### 1. 思考-执行分层：人从执行层全面退出

> "人类的主要职责是启动硅基、迭代硅基、实践硅基。"

所有既有工具都在优化"人如何更好地做"，FP 的主张是"人不必亲自做"。这是姿态上的根本差异——不是提效，是换位。

### 2. Project is Everything

- **Project 是系统的生命力**：Project 持续、高效地组织 Code，不断拓展系统能力边界，构建可持续演进的系统
- **Project 是 Code 的价值内核**：一切 Code 构建的智能，只有进入项目、形成闭环、解决真实问题，才拥有意义
- **Project 是 Code 有序聚焦的闭环**：落地、对齐业务、稳定运行、产生收益、形成壁垒

Todoist / Things / 滴答 的心智是 **Task 列表**，Notion / 飞书是 **自由数据库**。FP 的根基是**强制四层结构**（Project → Epic → US → Task），因为只有进入 Project 闭环的 Code 才有价值。Task 不是待办，是 Project 可被 AI 消化的最小指令单元。

### 3. Everything is Code 的操作化

笔记、流程、决策、交付——一切可复现逻辑都应被 Code 表达。FP 把这句哲学落成**执行端**：你不是在"做"任务，你是在"调度任务被 Code 做"。

### 4. 自迭代系统的执行端

**任意笔记系统（认知组织层）+ FocusPilot（执行调度层）= 一个持续自我重构的闭环**：**认知提升 × 密集做事 = 自我迭代**。FP 是让"密集做事"不再消耗人时间的关键齿轮。

> 闭环本身只要求笔记系统能持有标准化的 Markdown + frontmatter——任何系统都可承担"认知组织层"的角色。

## 五、市场洞察：范式错位

所有竞品都在错误的战场上优化错误的事。

| 范式               | 代表产品                | 帮用户解决什么     |
| ------------------ | ----------------------- | ------------------ |
| **列表范式**       | Todoist / Things / 滴答 | 帮你**记得做**     |
| **注意力范式**     | 番茄 Todo / Forest      | 帮你**专注做**     |
| **数据库范式**     | Notion / 飞书           | 帮你**组织着做**   |
| **调度范式（FP）** | FocusPilot              | 帮你**不必亲自做** |

**关键洞察**：2020 年前，"做事"是稀缺资源，任务管理工具的价值在"别漏"；2026 年 AI 执行成本趋零后，真正稀缺的是"方向判断 + 规划能力"，执行本身应该被外包给智能体。现有工具还把人当执行主体，这是代际错位。

这个判断有三条外部趋势支撑：

1. McKinsey 2025 年全球 AI 调研显示，Agent 已进入大量组织的实验阶段，但真正价值来自工作流重构，而不是单个 Agent 本身。
2. Gartner 2025 年预测，到 2027 年底超过 40% 的 Agentic AI 项目会因价值不清、成本和风险问题被取消，说明空泛 Agent 平台叙事会被快速淘汰。
3. Stack Overflow 2025 开发者调研显示，开发者广泛采用 AI 工具，但对 AI 输出准确性的信任不足，这意味着“上下文、状态、验收、可追责记录”会比单纯聊天能力更重要。

## 六、目标用户画像

**不是大众，是硬核前排**：

- **一人公司 / 独立开发者 / 研究型工作者**：不缺执行力，缺"把执行让出去"的能力
- **已经用 Claude Code / Codex 工作的 Vibe Coder**：熟悉 AI 调度，痛点是缺一个把项目-任务-AI 串起来的编排层
- **愿意把工作项目化的高密度执行者**：愿意接受"先用 Project/Epic/US 组织目标，再把 Task 交给 AI 执行"的认知成本

**不服务**：需要被提醒"今天记得买菜"的日常用户、追求任务管理美学的极简主义者。这不是这个产品的战场。

---

## 七、差异化护城河（三道壁）

1. **Markdown-Native 的开放协议**：FP 自己能生成和管理 Markdown workspace，同时也不绑定任何笔记应用；任何能读写 Markdown + frontmatter 的系统都可接入（Obsidian / Logseq / 纯文件夹 / 自建工具）。**契约属于 FP，存储属于用户**——这是反 SaaS 锁定的姿态，是赢得硬核用户的关键。
2. **AI 调度 ≠ AI 辅助**：Copilot / Cursor 是"AI 帮你写"，FP 是"AI 自己做完"——用户在更高一层抽象
3. **Project 层级强约束**：看似限制，实则是"让 AI 能自动接管"的前置条件——自由数据库天然不可被智能体可靠消化

---

## 八、暴露的关键张力（待回答）

1. **协议严格度 vs 接入门槛**：Markdown + 四层 frontmatter 的契约要严到何种程度？太严，用户难以采用；太松，AI 无法可靠消化。这是 FP 作为"协议引擎"的命脉。
2. **自然语言 → 可靠交付的边界**：哪些 Task 可以直接 AI 闭环、哪些必须人回路审核？这是产品可信度的命脉。
3. **反馈闭环的落地**：AI 产出如何回写到笔记、如何被人快速评审、如何沉淀到下一次规划——这条链路目前在 backlog，但它决定 FP 是"调度器"还是"真正的自迭代引擎"。
4. **自建 workspace vs 外部适配器**：FP 默认应先能自建和管理 Markdown workspace，再把 Obsidian / Logseq / 纯文件夹视为可选适配器。否则产品容易被误解为某个笔记系统的插件，而不是独立工具。
5. **多笔记系统适配策略**：先做 Obsidian / Logseq 各自的官方适配器，还是定义一个抽象的「笔记系统驱动接口」让社区贡献？前者快、后者远。
