# FocusPilot · 知行一体化个人自迭代系统产品定位

> **FocusPilot 核心定位：人与 AI 协同的知行一体化个人自迭代系统。**
>
> 采集信息，提炼规律，形成智慧，投入实践，内化能力。
>
> **Everything is Code. Project is Everything. System is Everything. FocusPilot runs the system.**
>
> 一切皆可 Code 化，一切都应 Project 化，一切价值都要沉淀为 System。FocusPilot 负责让这个个人系统持续运转。

## 一、顶层理念：Code / Project / System / FocusPilot

FocusPilot 的产品理念不是单点功能，而是一组层层递进的系统命题。

### 1. Everything is Code

**一切可复现、可编排、可执行、可迁移的逻辑，最终都可以 Code 化。**

这里的 Code 不只指程序代码，也包括 Prompt、Agent 工作流、RAG 链路、信息加工流程、知识卡片、认知规律、任务拆解规则和实践 SOP。

在认知系统里，Code 的最小颗粒是**认知规律**：从信息和经验中提炼出来，可遵循、可复用、可迁移。
在实践系统里，Code 的落地形态是可执行流程：Project、Task、Agent 指令、自动化脚本和交付产物。

### 2. Project is Everything / Everything is Project

**一切值得投入注意力的认知与实践，都应当 Project 化。**

Project 不是简单任务容器，而是个人系统中的价值组织单元。认知类 Project 负责吸收高质量信息、沉淀知识、形成判断力；实践类 Project 负责把判断力投入真实场景，形成交付、收入、影响力和现实验证。

没有 Project，信息只是碎片，知识只是收藏，行动只是消耗。进入 Project，信息、知识、任务、产物和反馈才会形成闭环。

### 3. System is Everything

**单个 Project 不是终点，持续自迭代的 System 才是终点。**

Project 让一次行动产生价值，System 让多次行动形成复利。这个 System 不是 FocusPilot 这个 App 本身，而是用户自己的个人自迭代系统：由认知系统、实践系统、Markdown 工作记忆、Agent 执行和人类判断共同构成。

它的核心循环是：

```text
高质量信息
  -> 认知规律
  -> 智慧体系
  -> 实践项目
  -> 现实结果
  -> 人类判断
  -> 新一轮信息采集与系统调整
```

### 4. FocusPilot runs the system

**FocusPilot 不是替代人的系统，而是运行个人自迭代系统的 AI 工作层。**

FocusPilot 负责高密度采集、加工、输出和执行；人负责规律判断、记忆内化、实践决策和方向修正。它把 Code、Project、System 串起来，让个人系统不再停留在笔记、愿望或零散任务里，而是持续运转。

一句话：

> **Code 化规律，Project 化行动，System 化成长，FocusPilot 让你的个人系统持续自迭代。**

## 二、核心定位

**FocusPilot 不是任务管理工具，也不是单纯的 AI 执行器，而是把信息转化为能力的个人自迭代系统。**

它解决的核心问题不是“如何记住更多任务”，而是“如何让高质量信息进入认知系统，让认知规律形成智慧，再通过真实项目实践转化为个人能力”。

这个系统由两部分协同运转：

1. **FocusPilot 负责采集、加工与项目化输出**
   它汇聚 to-read、灵感、stickies、材料、任务和外部高质量信息源，按 Project 组织加工，生成 Report、知识卡片、执行任务或实际产物。

2. **人类负责记忆、实践与判断**
   人通过记忆卡片、费曼学习法、间隔重复等方式内化认知规律，并在真实场景中实践智慧。实践结果由人自主判断，再决定是否扩大信息源、调整采集范围、修改系统配置或开启新一轮项目。

因此，FocusPilot 的自迭代不是“AI 全自动替人进化”，而是：

> **AI 负责高密度采集、加工、输出和执行；人负责规律判断、记忆内化、实践决策和方向修正。**

核心公式：

> **高质量信息 × 知识加工 × 智慧实践 × 人类判断 = 个人能力增长。**

## 三、核心概念

### 1. 认知规律

**认知规律是最小单元的 Code 化客观逻辑。**

它来自信息、经验和实践，是被提炼出来的可遵循、可复用、可迁移的底层规律。认知规律以知识卡片为主要载体落地存储，是构成智慧的基础颗粒。

任何事情都要遵循规律，认知与实践也不例外。FocusPilot 的知识加工，本质上不是保存信息，而是帮助人从信息中提炼可复用的认知规律。

### 2. 智慧

**智慧是成体系的认知规律集合。**

当多条相关认知规律被结构化串联，就形成完整的判断框架，能指导人在不同场景下判断“做什么、怎么做、优先级如何”。智慧不是零散知识，而是能指导决策和行动的规律体系。

### 3. 能力

**能力是个体依托体系化认知规律，在真实场景中持续遵循规律、解决问题、落地目标的稳定行动力。**

能力不是知道很多规律，而是能熟练运用规律。它需要经过反复实践验证，最终沉淀为可复用、可泛化的综合实战本领。

### 4. 实践到能力的转化路径

```text
单次实践执行
  -> 结果复盘
  -> 剔除无效动作
  -> 提炼新的认知规律
  -> 内化为标准化知识卡片
  -> 织入既有智慧体系
  -> 依托智慧持续指导实践
  -> 形成稳定解决问题的综合能力
```

## 四、全链路流转逻辑

FocusPilot 的完整链路不是 `Project -> Task -> Execute`，而是：

```text
高质量信息采集
  -> Intake
  -> Project 化加工
  -> Output
  -> 记忆内化
  -> 智慧实践
  -> 人类判断
  -> 调整采集范围与系统配置
  -> 新一轮自迭代
```

### 1. 信息采集

人类决定信息采集方向、关注主题、信息源质量和系统配置。FocusPilot 负责把 to-read、灵感、stickies、外部材料、行业资讯、公众号、视频平台等信息归集到 intake。

高质量信息采集系统目前可以作为后续能力扩展，但产品理念上必须预留它的位置：Intake 是个人自迭代系统的统一入口，不只是待办收件箱。

### 2. 处理加工

FocusPilot 按 Project 组织信息加工，将原始信息结构化为：

- Report：阶段性研究报告、分析报告、整合报告；
- 知识卡片：承载认知规律的最小记忆单元；
- Task：需要进入实践系统的可执行事项；
- 实际产物：代码、文档、视频、公众号文章、方案、交付材料。

人类负责 Review，判断信息是否有价值、规律提炼是否准确、产物是否可用。

### 3. 记忆内化

知识卡片不是最终目标。它们只是帮助人记忆认知规律的载体。

人通过费曼学习法、间隔重复、主动回忆和实践验证，把认知规律织入自己的智慧体系。

### 4. 实践落地

人依托智慧进行实践决策，在 FocusPilot 中创建或收编项目，把实践目标拆成 Project / Epic / US / Task。FocusPilot 辅助完成任务拆解、Agent 调度、内容创作、代码生成、报告产出等落地工作。

### 5. 迭代反哺

实践产生新问题、新经验和新判断。是否需要继续采集信息、扩大信息源、修改项目方向、更新系统配置，由人做决策。FocusPilot 再根据新的方向启动下一轮采集、加工、输出和项目执行。

## 五、认知系统与实践系统

FocusPilot 的个人自迭代由两个系统共同驱动：**认知系统**和**实践系统**。

### 1. 认知系统

认知系统通过知识项目运转。

知识项目负责吸收高质量信息、沉淀知识、生成记忆卡片、形成判断框架。它的目标不是直接交付商业结果，而是提升人的判断力，让人持续“看得更准”。

典型项目包括：

- AI 趋势研究；
- 市场洞察与商业模式；
- 技术路线研究；
- 个人方法论；
- 某个长期学习或认知主题。

一句话：

> **认知系统把信息转化为判断力。**

### 2. 实践系统

实践系统通过实践项目运转。

实践项目包括工作项目、收入项目和创作项目。它的目标是把判断力投入真实世界，形成交付、收入、影响力和现实验证，让人持续“做得更成”。

典型项目包括：

- 工作交付项目；
- 客户或组织内项目；
- 视频生成项目；
- 公众号项目；
- 产品开发项目；
- 自动化工具项目；
- 商业实验项目。

一句话：

> **实践系统把判断力转化为现实结果。**

### 3. 双系统循环

认知系统和实践系统不是上下级关系，而是平等协同的双循环：

```text
认知系统
  -> 生产判断力、方法论、认知规律
  -> 指导实践系统

实践系统
  -> 生产交付、收入、影响力、现实反馈
  -> 触发人类判断
  -> 反向更新信息采集方向与认知系统
```

更凝练地说：

> **认知系统生产判断力，实践系统生产现实结果，人类判断把现实结果转化为下一轮信息采集、规律修正和能力增长。**

## 六、人与 AI 的协同分工

FocusPilot 的核心不是让 AI 替代人，而是重新划分人与 AI 在个人自迭代系统中的职责。

| 环节     | 人类负责                             | FocusPilot / AI 负责                                   |
| -------- | ------------------------------------ | ------------------------------------------------------ |
| 信息采集 | 决定方向、主题、信息源质量标准       | 自动归集 to-read、灵感、stickies、外部资料和高质量信息 |
| 加工处理 | Review 产物，判断规律提炼是否准确    | 结构化梳理、报告撰写、知识卡片生成、任务拆解           |
| 规律沉淀 | 记忆、理解、费曼复述、间隔复习       | 维护卡片、关联来源、提示复习、组织知识结构             |
| 实践落地 | 做核心判断、授权执行、验收结果       | 调度 Agent、生成内容、执行任务、产出交付物             |
| 迭代反哺 | 判断是否扩大采集、修正方向、调整配置 | 根据新方向重新采集、加工、输出和项目化执行             |

人类负责“规律把控”和“实践决策”，FocusPilot 负责“信息处理”和“执行放大”。两者共同构成知行一体化的自迭代循环。

## 七、Markdown 工作记忆

FocusPilot 真正依赖的不是某个笔记应用，而是一份开放的 Markdown + frontmatter 契约。

如果用户没有现成笔记系统，FocusPilot 应该能自己创建和管理一个 Markdown workspace；如果用户已有 Obsidian / Logseq / 纯文件夹 / 自建工具，FocusPilot 则通过同一份 Markdown 契约接入。

换句话说：

> **笔记系统是可替换外壳，Markdown 工作记忆才是核心资产。**

## 八、架构抽象：个人 AI 工作层

从用户价值上看，FocusPilot 是个人自迭代系统；从架构抽象上看，它具备个人 AIOS 的结构。它不是底层操作系统，而是在个人工作层提供类似操作系统的关键能力：记忆介质、任务调度、执行进程、权限确认、系统日志、工作入口和反馈循环。

| AIOS 抽象 | FocusPilot 对应                                            |
| --------- | ---------------------------------------------------------- |
| 文件系统  | Markdown workspace                                         |
| 进程      | Agent 执行会话                                             |
| 调度器    | Task 状态流转与执行队列                                    |
| Shell     | Dashboard / Quick Panel / 对话面板                         |
| 应用      | Claude Code / Codex / Gemini / MCP 工具                    |
| 权限系统  | 人工确认、验收、回滚                                       |
| 系统日志  | Execution Log / Reports                                    |
| 长期记忆  | Project notes / Markdown frontmatter / Knowledge artifacts |
| 桌面环境  | 今日聚焦、窗口管理、番茄钟、专注入口                       |

`AIOS` 不应作为唯一对外主词，而应作为架构解释：FocusPilot 对外是个人自迭代系统，对内是运行这个系统的 AI 工作操作层。

## 九、市场洞察：范式错位

所有竞品都在错误的战场上优化错误的事。

| 范式               | 代表产品                | 帮用户解决什么       |
| ------------------ | ----------------------- | -------------------- |
| **列表范式**       | Todoist / Things / 滴答 | 帮你记得做           |
| **注意力范式**     | 番茄 Todo / Forest      | 帮你专注做           |
| **数据库范式**     | Notion / 飞书           | 帮你组织着做         |
| **调度范式（FP）** | FocusPilot              | 帮你把信息转化为能力 |

**关键洞察**：2020 年前，“做事”是稀缺资源，任务管理工具的价值在“别漏”；2026 年 AI 执行成本趋零后，真正稀缺的是“方向判断 + 规律把控 + 实践决策”。现有工具仍把人当执行主体，这是代际错位。

这个判断有三条外部趋势支撑：

1. McKinsey 2025 年全球 AI 调研显示，Agent 已进入大量组织的实验阶段，但真正价值来自工作流重构，而不是单个 Agent 本身。
2. Gartner 2025 年预测，到 2027 年底超过 40% 的 Agentic AI 项目会因价值不清、成本和风险问题被取消，说明空泛 Agent 平台叙事会被快速淘汰。
3. Stack Overflow 2025 开发者调研显示，开发者广泛采用 AI 工具，但对 AI 输出准确性的信任不足，这意味着“上下文、状态、验收、可追责记录”会比单纯聊天能力更重要。

## 十、目标用户画像

**不是大众，是硬核前排**：

- **一人公司 / 独立开发者 / 研究型工作者**：不缺执行力，缺“把执行让出去、把知识沉淀下来”的系统；
- **已经用 Claude Code / Codex 工作的 Vibe Coder**：熟悉 AI 调度，痛点是缺一个把项目、任务、AI、知识和反馈串起来的编排层；
- **愿意把工作项目化的高密度执行者**：愿意接受“先用 Project 组织认知与实践，再把 Task 交给 AI 执行”的认知成本。

## 十一、差异化护城河

1. **Markdown-Native 的开放协议**：FocusPilot 自己能生成和管理 Markdown workspace，同时也不绑定任何笔记应用；任何能读写 Markdown + frontmatter 的系统都可接入。
2. **认知加工 + 实践执行一体化**：它不只是执行任务，也负责把高质量信息加工成知识、卡片、报告和实践项目。
3. **AI 调度 ≠ AI 辅助**：Copilot / Cursor 是“AI 帮你写”，FocusPilot 是“AI 承接执行，人类把控规律与判断”。
4. **Project 层级强约束**：看似限制，实则是让信息、知识、实践和产物能被 AI 稳定消化的前置条件。

## 十二、暴露的关键张力（待回答）

1. **协议严格度 vs 接入门槛**：Markdown + frontmatter 的契约要严到何种程度？太严，用户难以采用；太松，AI 无法可靠消化。
2. **信息采集自动化范围**：to-read、灵感、stickies、行业资讯、公众号、视频平台等入口，哪些应优先进入 V1？
3. **知识卡片质量控制**：认知规律的提炼是否准确，需要怎样的人类 Review 与记忆机制？
4. **实践项目的验收边界**：哪些 Task 可以直接 AI 闭环，哪些必须由人回路审核？
5. **自建 workspace vs 外部适配器**：FocusPilot 默认应先能自建和管理 Markdown workspace，再把 Obsidian / Logseq / 纯文件夹视为可选适配器。
