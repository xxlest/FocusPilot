# MVP 范围定义

## 一、文档目的

本文档用于明确项目第一阶段 MVP 的边界，避免在早期阶段出现以下问题：

- 一开始想做太多
- 路线图过大，迟迟无法落地
- 讨论很多，但没有最小闭环
- 客户验证和产品开发不同步

MVP 的目标不是“做一个完整系统”，而是：

**用最小可用的功能闭环，验证客户是否真的愿意使用、愿意继续推进，甚至愿意付费。**

## 二、MVP 的基本原则

### 1. 先验证价值，不先追求完整

MVP 要回答的核心问题只有一个：

- 客户是否会因为这个最小闭环感受到真实价值

而不是：

- 系统是否已经功能齐全

### 2. 先解决一个高价值场景

不要试图一开始同时覆盖：

- 晨会
- 督办
- 渠道复盘
- 客户抗性
- 管理层总览
- 策略建议

必须优先选一个最有感知、最容易验证、最容易接数据的场景。

### 3. 先做“人机协同 MVP”，不要一开始做“全自动系统”

早期更适合：

- AI 先给出初步分析
- 人工补充判断
- 最终形成可交付结果

而不是一开始追求：

- 全自动判断
- 全自动决策
- 全自动跨系统联动

## 三、MVP 建议切入点

基于当前讨论，更推荐的 MVP 不是“项目 PM 全场景助手”，而是聚焦以下其中一个场景。

### 方案 A：经营总览 / 周报生成 MVP

这是最推荐的起步方式。

核心价值：

- 帮管理层或项目负责人快速看清盘面
- 让客户更容易直接感知结果

最小闭环包括：

1. 导入基础业务数据
2. 自动生成关键经营指标汇总
3. 输出一版结构化周报/经营摘要
4. 支持人工补充说明和修订

为什么适合：

- 数据接入相对简单
- 结果易展示
- 管理层和业务方都能看懂

### 方案 B：多项目经营分析 MVP

如果目标客户是代建公司或代理销售公司，这个场景价值更高。

核心价值：

- 帮管理者同时看多个项目
- 找出异常项目和重点项目

最小闭环包括：

1. 导入多个项目的关键指标
2. 排序显示项目健康度
3. 输出异常项目提示
4. 生成简单经营判断

为什么适合：

- 更符合“多项目管理”特征
- 更容易形成管理层视角差异化

### 方案 C：项目经营分析 + 汇报辅助 MVP

适合想先从项目 PM 真实工作流切入时使用。

核心价值：

- 帮项目负责人减少经营分析和汇报准备时间

最小闭环包括：

1. 导入项目经营数据
2. 自动生成本周经营变化摘要
3. 输出汇报版草稿
4. 支持人工改写和确认

## 四、当前最建议的 MVP 选择

结合前面的讨论，当前更建议优先做：

### 优先选择：方案 A + 方案 B 的简化结合

即：

**“多项目经营总览 + 周报生成”**

这是当前最适合代建公司、代理销售公司以及部分管理层场景的最小切入点。

原因：

1. 它比单纯 PM 工具更接近买单人
2. 它比复杂策略系统更容易先落地
3. 它比大而全平台更容易快速验证

## 五、MVP 不做什么

为了避免失控，第一阶段明确不做以下内容：

### 1. 不做完整 CRM

不做：

- 客户录入
- 线索管理
- 完整销售漏斗系统

### 2. 不做复杂审批流

不做：

- 复杂流程配置
- 多角色审批链路
- 组织级协同平台

### 3. 不做大而全 BI 平台

不做：

- 通用大屏
- 任意维度自定义报表平台
- 重型数据中台

### 4. 不做全自动决策

不做：

- 自动给出最终经营决策
- 自动调整预算
- 自动改策略并下发执行

### 5. 不做过多 Agent 并发

早期先避免一开始做：

- 6 个以上独立 Agent
- 多条复杂工作流
- 高频定时自动推送体系

## 六、MVP 最小功能清单

建议 MVP 先只包含以下功能。

### 1. 数据导入

支持：

- Excel
- CSV
- 标准表格导入

目标：

- 能拿真实客户样本先跑起来

### 2. 核心指标计算

至少支持：

- 来访
- 认购
- 签约
- 回款
- 渠道贡献
- 关键异常变化

### 3. 摘要生成

系统自动输出：

- 本周/本月关键变化
- 异常项目或异常指标
- 初步经营判断

### 4. 报告输出

至少支持：

- 管理层摘要版
- 项目经营分析版

### 5. 人工补充与修订

支持人工：

- 补充原因
- 修改口径
- 增加判断

这一步非常关键，MVP 阶段必须保留。

## 七、MVP 成功标准

MVP 是否成立，不看功能数量，而看以下几个标准：

### 1. 客户是否愿意用真实数据跑

如果客户连数据都不愿意给，MVP 价值没有被真正验证。

### 2. 客户是否觉得结果有用

不是说“挺好”，而是：

- 这东西能替代我一部分原来工作
- 这东西让汇报和判断更快

### 3. 客户是否愿意继续推进

例如：

- 再约下一次
- 再跑一次
- 扩到第二个项目

### 4. 客户是否愿意讨论试点或费用

一旦进入这一步，MVP 就开始从“验证产品”进入“验证商业”。

## 八、MVP 第一阶段建议产出

建议第一阶段只追求以下交付物：

1. 一版可导入真实表格的原型
2. 一版经营总览输出
3. 一版报告生成结果
4. 一版人工修订流程

## 九、MVP 阶段的判断标准

如果 MVP 开发过程中出现以下现象，应立即警惕：

- 需求不断扩散
- 每个客户都想要完全不同功能
- 团队大部分时间都在讨论系统架构而不是验证场景
- 没有人愿意用现有最小版本试

这些通常意味着：

- MVP 边界失控
- 产品还没有收敛

## 十、最终建议

当前阶段最适合的 MVP，不是“房地产项目全能操盘平台”，而是：

**一个能接入真实数据、输出多项目经营总览和结构化周报的人机协同型最小闭环。**

一句话总结：

**先证明你能帮客户看清楚，再证明你能帮客户做决定；先做经营总览，再谈完整操盘系统。**
