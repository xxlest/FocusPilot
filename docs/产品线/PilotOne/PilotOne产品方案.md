# PilotOne 产品方案

> **定位**：面向中小企业 AI 化的企业智能操作系统。
> **目标**：赋能万家中小企业拥有自己的 AI 组织能力。

---

## 1. 产品定位

**PilotOne 是中小企业 AI 化操作系统：以行业本体和企业实例本体为业务底座，以 Hermes-Agent 自迭代引擎为技术内核，以 PilotOne-Agent 数字员工体系为企业场景产品化形态，把战略、客户、订单、产品、交付、渠道、知识和流程统一建模，让 AI 持续驱动企业经营闭环。**

它不是一个聊天机器人，也不是单点自动化工具，而是一套帮助中小企业完成 AI 化升级的组织系统。

### 1.1 核心问题

中小企业并不缺单个软件，真正缺的是一套能把企业业务跑起来的智能组织能力：

- 老板掌握太多隐性经验，员工和系统无法复用。
- ERP、CRM、财务、IM、表格和文档分散，数据口径不统一。
- 流程靠人催，异常靠人发现，复盘靠人记忆。
- AI 工具只能回答问题，无法持续理解公司、执行任务和沉淀经验。
- 企业没有足够预算和团队建设完整数字化体系。

PilotOne 的机会在于：用 AI 原生方式补齐中小企业“数字化 + 自动化 + 组织知识沉淀”的缺口。

### 1.2 产品命题

PilotOne 的产品命题可以概括为三句话：

1. **让 AI 理解企业**：通过行业本体定义可复用的业务概念和语义关系，再通过企业实例本体承载每家公司的私有数据。
2. **让 AI 执行业务**：通过 PilotOne-Agent 数字员工体系，把 Hermes-Agent 的自迭代能力产品化为有职责、有状态、有权限、有 Routine、有事件输出的企业成员。
3. **让企业持续进化**：通过 OrgBrain 记录执行事件、交付物、经验和规则，让 Routine、SOP、规则和本体结构进入分层自治的持续优化。

### 1.3 目标客户

PilotOne 优先面向 10-200 人中小企业，尤其是以下类型：

| 企业特征 | 典型痛点 |
|----------|----------|
| 老板强驱动 | 决策和经验集中在老板，员工执行靠催促 |
| 系统碎片化 | ERP、CRM、财务、IM、表格、文档分散，数据无法连起来 |
| 重复流程多 | 客户跟进、报价、交付、对账、内容获客存在大量重复劳动 |
| 数字化团队弱 | 没有专职 IT、数据团队或自动化工程师 |
| 业务有明确 SOP | 流程相对稳定，适合数字员工逐步接管 |

不优先服务的客户：

- 组织复杂、合规要求极高的大型集团。
- 业务高度非标准化、流程不可复用的早期探索团队。
- 只想买一个聊天机器人或 AI 写作工具的轻量用户。

### 1.4 核心价值

给老板：

- 从“盯人、催事、找数据”转向“设目标、批规则、验结果”。
- 每天看到关键经营异常、进度阻塞和决策建议。
- 把个人经验逐步沉淀为企业可复用的规则和 SOP。

给员工：

- 减少信息搬运、重复整理、跨系统查询和机械汇报。
- 让员工专注在专业判断、客户关系、现场执行和复杂沟通。
- 数字员工先产出草稿、建议和执行结果，人类做确认与修正。

给企业：

- 业务流程更稳定，经营信息更透明。
- 经验可以沉淀，规则可以复用，组织能力可以持续累积。
- 以渐进式方式完成 AI 化，而不是一次性替换现有系统。

### 1.5 核心循环

```text
经营目标
  → 企业本体建模
  → PilotOne-Agent 数字员工执行
  → 业务事件和交付物沉淀
  → OrgBrain 归档经验和规则
  → Hermes-Agent 判断自治级别
  → 自动生效 / 通知负责人 / 人工审批
  → 下一轮执行自动使用新规则
```

这个循环的关键不是“AI 生成内容”，而是“AI 参与企业经营后的每次执行都会让企业系统变得更懂自己”。

### 1.6 规模化机制

PilotOne 要赋能万家中小企业，不能依赖每家企业从零咨询交付。规模化的关键是把“行业共性”沉淀成模板，把“企业差异”留在实例层，把“跨企业经验”脱敏后反哺行业模板。

| 机制 | 说明 | 目标 |
|------|------|------|
| 行业本体模板 | 先选择少数高频行业预置概念层、关系层、默认规则、Routine 和数字员工配置 | 新企业不从空白系统开始，而是从行业默认操作系统开始 |
| 场景包模板 | 把客户跟进、经营日报、报价预警、内容获客、交付催办等高频场景打包 | 中小企业按问题购买，不按抽象模块购买 |
| 对话式企业入驻 | 通过老板访谈、样表导入、系统授权和样例校准生成企业实例本体 | 30 分钟完成首个场景包初始化，1-3 天完成关键系统接入和本体校准，2-4 周形成第一轮自迭代规则 |
| 跨企业经验网络 | 同行业企业的脱敏 SOP、规则建议、Routine 参数和异常模式回流到行业模板 | 第 100 家企业继承前 99 家的共性经验，但不泄露任何企业私有数据 |

优先行业不应一开始泛化到所有中小企业。更务实的路径是先从制造/贸易/广告营销/本地服务中选择 1-2 个高重复、强流程、数据对象明确的行业切入，跑通“模板 → 入驻 → 执行 → 自迭代 → 模板升级”的闭环后再横向扩展。

## 2. PilotOne-Agent 数字员工体系

> **命名说明**：Hermes-Agent 是底层自迭代引擎和方法论，PilotOne-Agent 是 Hermes-Agent 在企业 AI 化场景中的产品化实例。对外使用 PilotOne-Agent 统一产品心智，对内保留 Hermes-Agent 作为引擎层命名。

**PilotOne-Agent = Hermes-Agent 自迭代引擎 + 角色 Profile + 行业/企业本体上下文 + Tool 权限 + Skill 流程 + Routine 任务 + 审批边界 + 事件输出 + 自复盘机制。**

它不是一个“聊天角色”，而是一个能围绕企业职能持续工作的数字员工。

每个数字员工都应该回答清楚七个问题：

| 问题 | 对应机制 |
|------|----------|
| 我是谁 | Profile / Identity |
| 我负责什么 | Role scope / 职责边界 |
| 我能看什么 | 本体论查询权限 / 数据权限 |
| 我能做什么 | Tool 权限 / API 权限 |
| 我怎么做 | Skill / SOP / Routine |
| 我什么时候需要人确认 | 审批规则 / 风险等级 / 授权边界 |
| 我如何变得更好 | Events / Rule Proposal / Hermes 分层自治 |

### 2.1 五类通用数字员工

数字员工不是“高级 Prompt 模板”。每个数字员工都绑定本体对象、持久状态、工具权限、事件输出和协作下游。

| 数字员工 | 绑定本体对象 | 状态持久性 | 事件输出 | 协作下游 |
|----------|--------------|------------|----------|----------|
| 战略参谋 | Customer / Order / Product / Financial / Supplier / Event / Rule | 记住历史经营建议、老板偏好、关键指标阈值和上轮决策结果 | `risk.detected`、`goal.decomposed`、`rule.proposed` | 运营专家、获客专家、知识官 |
| 运营专家 | Order / Project / Product / SKU / BOM / Supplier / Contract / Task | 记住交付状态、供应商表现、异常处理记录和排产偏好 | `delivery.blocked`、`supplier.price_changed`、`task.assigned` | 战略参谋、客户专家、知识官 |
| 获客专家 | Channel / Campaign / Lead / Asset / Customer / Product | 记住渠道表现、内容风格、线索质量和转化数据 | `lead.created`、`campaign.reviewed`、`content.published` | 客户专家、战略参谋、知识官 |
| 客户专家 | Customer / Contact / Order / Contract / Receivable / Event | 记住客户偏好、历史承诺、回访节奏、投诉和复购窗口 | `customer.silent`、`renewal.due`、`quote.requested` | 运营专家、获客专家、知识官 |
| 知识官 | Event / Rule / SOP / Asset / Routine | 记住规则变更历史、SOP 版本、误判案例和最佳实践 | `sop.proposed`、`routine.adjusted`、`case.archived` | 所有数字员工 |

第 1 个月，数字员工主要依赖行业模板和人工确认；第 3 个月，开始形成企业私有规则和客户/供应商偏好；第 6 个月，Routine 参数、SOP 步骤和预警阈值应明显企业化，能主动发现重复异常并提出规则更新。

### 2.2 8 角色方案到 5 角色产品抽象的映射

`企业AI数字员工整体方案.md` 中的 8 角色是中型企业理想终态，PilotOne 产品方案中的 5 角色是更适合首批产品化和中小企业采购的抽象。裁剪原则是：先按业务闭环合并角色，再在行业场景包中拆出专属子角色。

| 8 角色方案 | PilotOne 5 角色抽象 | 处理理由 |
|------------|---------------------|----------|
| COO 调度 | 战略参谋 + 运营专家 | 调度能力拆到目标拆解和执行推进，不单独成为首批付费角色 |
| 秘书汇报 | 战略参谋 + 知识官 | 汇报是经营洞察和事件归档的表现，不独立成角色 |
| 产品研发 | 运营专家 | 对非软件企业先归入交付/项目推进，软件行业可作为行业子角色扩展 |
| 项目管理 | 运营专家 | 归入订单、项目、任务和交付闭环 |
| 商务销售 | 客户专家 | 销售跟进、报价、回款和客户维护归入客户经营闭环 |
| 市场内容 | 获客专家 | 内容生产、渠道分发和线索培育归入获客闭环 |
| 人事行政 | 知识官 + 运营专家 | 首批不作为核心差异化角色，先服务 SOP、行政流程和内部执行 |
| 财务法务 | 战略参谋 + 运营专家 | 财务分析归战略，合同/对账/回款归运营；复杂法务作为后续扩展 |

### 2.3 数字员工的三类产物

| 产物类型 | 内容 | 价值 |
|----------|------|------|
| 交付物 | 报价单、合同草案、PRD、技术方案、日报、周报、经营分析报告 | 直接服务客户、同事或管理者 |
| 累积资产 | 客户画像、知识库、事件流水、工作日志、案例库 | 形成企业可复用的业务记忆 |
| 自迭代产物 | SOP 改进建议、效率指标、审批规则建议、Routine 调整建议、本体关系修正建议 | 让数字员工下次做得更好 |

### 2.4 人机协作模式

PilotOne-Agent 中，AI 负责高频、可结构化、可复盘的执行工作；人类负责目标、判断、授权和验收。

```text
[人] 老板设定经营目标
  → [AI] 战略参谋拆解为业务任务
  → [AI] 运营/获客/客户专家执行任务
  → [人] 对关键方向、资金、对外内容进行确认
  → [AI] 写回业务系统并记录事件
  → [AI] 知识官归档经验并提出规则更新
  → [系统/人] 按自治级别自动生效、通知或审批
  → [AI] 下次执行使用新规则
```

人类主要在三种场景介入：

- **决策确认**：方向是否正确，方案是否符合老板意图。
- **信息补充**：AI 无法从系统中读取的事实、背景和约束。
- **审批授权**：涉及资金、合同、对外发送、客户承诺和规则更新。

### 2.5 Routine 机制

数字员工不是等用户提问才工作。每个数字员工都应该拥有自己的 Routine，由定时任务、事件触发和审批回调驱动。

| 数字员工 | Routine 示例 |
|----------|--------------|
| 战略参谋 | 每天早上扫描经营指标，生成今日重点和异常提醒 |
| 运营专家 | 每天检查交付进度、超时任务和待对账订单 |
| 获客专家 | 每天扫描新线索、生成内容草稿、复盘内容数据 |
| 客户专家 | 每天检查沉默客户、待回访客户和即将复购客户 |
| 知识官 | 每周扫描事件和工作日志，提出 SOP 改进建议 |

Routine 应采用两阶段执行：

1. **轻量检测**：先用 SQL/API 检查是否有变化，不调用大模型。
2. **完整执行**：确实有变化时才创建 Agent，加载上下文并生成有价值输出。

这样可以降低成本、减少噪音，并保证数字员工只在有价值的时刻打扰人。

### 2.6 Routine、Events、Rule 的微观闭环

Routine 不只是定时任务，Events 也不只是日志。二者必须通过 Rule 形成 Hermes 自迭代的微观闭环。

```text
Routine: 每天检查沉默客户
  → Event: 标记客户 A 为沉默客户
  → Result: 客户经理反馈“旺季客户本来 60 天才联系一次，此次为误判”
  → Rule Proposal: 对该行业旺季客户，把沉默阈值从 30 天调整为 60 天
  → 自治级别判断: L1 SOP/参数优化，生效并通知客户负责人
  → Routine: 下次使用新阈值
```

这条闭环让数字员工不是“按固定脚本运行”，而是把每次误判、成功和异常都转化为下一轮 Routine、SOP 或规则的改进输入。

### 2.7 Hermes 分层自治

单一“全部人工审批”的自迭代会带来审批疲劳，也会拖慢系统进化。PilotOne 采用 Hermes-Agent 的分层自治原则：

```text
执行 SOP / Routine
  → 记录 Events、结果反馈和误判样本
  → Hermes-Agent 复盘异常、成功模式和效率瓶颈
  → 生成参数 / SOP / 规则 / 本体结构变更建议
  → 判断自治级别 L0-L3
  → 自动生效 / 通知负责人 / 业务负责人确认 / 老板或管理员决策
  → 写回 Rule / SOP / Routine / Ontology
  → 下一轮执行使用新版本
```

| 层级 | 变更类型 | 示例 | 生效策略 |
|------|----------|------|----------|
| L0 参数微调 | 不改变业务规则，只调整执行参数 | 邮件发送时间从 09:00 调到 10:00；提醒频率从每天改为隔天 | 自动生效，记录事件 |
| L1 SOP 步骤优化 | 不改变责任和审批边界的小流程优化 | 先查库存再生成报价；客户回访前自动附上最近订单摘要 | 自动生效并通知负责人 |
| L2 业务规则变更 | 影响价格、客户分级、风控、对外承诺 | A 类客户账期从 30 天延长到 45 天；报价折扣阈值调整 | 需要业务负责人确认 |
| L3 本体结构变更 | 改变对象、关系、权限或跨系统语义 | 新增“渠道”对象；把“项目”拆成“订单项目”和“内部项目” | 需要老板/管理员决策 |

所有层级都必须可追踪、可回滚、可解释。区别只在于是否需要人类事前审批。

## 3. 企业本体论与 OrgBrain

企业本体论负责把企业里的关键对象、关系、事件和规则变成 AI 可理解、可查询、可更新的结构。OrgBrain 负责沉淀企业的业务记忆，包括执行记录、经验、规则和决策参考。

### 3.1 为什么需要企业本体论

中小企业 AI 化的瓶颈通常不是模型能力，而是企业上下文不可用：

- 同一个客户在 CRM、财务、合同和微信群里的名称不一致。
- 订单、项目、报价、回款、交付和客户关系散落在不同系统。
- 老板和老员工知道的业务规则没有结构化记录。
- AI 每次执行都像第一次了解公司，无法继承历史经验。

### 3.2 三层本体架构

PilotOne 的本体论不能停留在“列实体表”。它需要区分行业共性、企业私有实例和可执行推理。

| 层级 | 作用 | 制造业示例 |
|------|------|------------|
| 行业本体层 | 定义某个行业可复用的概念体系和语义关系 | 制造业共用“客户需求 → 产品/服务 → SKU/BOM → 工序 → 供应商 → 成本 → 报价 → 订单 → 交付” |
| 企业实例层 | 承载每家企业自己的客户、SKU、供应商、价格、合同、员工、渠道和历史事件 | 五金厂有螺丝、冲压件、镀锌工序；注塑厂有模具、塑料粒子、注塑工序 |
| 推理与迁移层 | 基于概念关系、企业实例和事件流进行多跳推理，并把脱敏经验沉淀回行业模板 | 供应商涨价影响 BOM，BOM 影响产品成本，产品成本影响客户报价，系统触发毛利预警 |

迁移能力来自这个分层：五金厂和注塑厂共享制造业的概念层，但各自的 SKU、供应商、工艺参数和报价规则属于企业实例层。新企业入驻时继承行业概念和默认 Routine，只需要校准自己的实例数据和少量规则。

### 3.3 核心对象

PilotOne 第一阶段不需要追求完整复杂的知识图谱，先围绕经营闭环定义最小可用本体。

| 对象 | 说明 |
|------|------|
| Customer 客户 | 客户主体、联系人、偏好、分层、历史互动 |
| Contact 联系人 | 客户侧联系人、角色、联系方式、关系强度、沟通偏好 |
| Product / Service 产品或服务 | 企业对外销售的产品、服务、报价项和交付承诺 |
| SKU / BOM 物料与成本结构 | SKU、物料清单、工序、成本项、供应商依赖 |
| Process 工序 | 生产、交付或服务流程中的关键步骤、产能、耗时和质量要求 |
| Channel 渠道 | 抖音、小红书、公众号、私域、招投标平台、线下渠道 |
| Campaign 营销活动 | 内容主题、投放渠道、线索来源、转化效果 |
| Lead 线索 | 潜在客户、来源渠道、意向等级、跟进状态 |
| Order 订单 | 订单金额、状态、交付周期、关联客户和项目 |
| Project 项目 | 项目目标、阶段、任务、负责人、交付物 |
| Task 任务 | 数字员工或人类员工需要执行的最小工作单元 |
| Contract 合同 | 合同金额、条款、账期、续约和风险信息 |
| Receivable 应收 | 应收金额、账期、到期日、回款状态、催收记录 |
| Financial 财务指标 | 营收、成本、毛利、现金流、费用、预算和经营指标 |
| Supplier 供应商 | 价格、交付、质量、历史合作记录 |
| Employee / DigitalEmployee 员工 | 人类员工和数字员工的角色、职责、权限、可审批事项 |
| Asset 资产 | 文档、报价模板、内容素材、案例、图片、视频、知识材料 |
| SOP / Routine 流程 | 标准作业流程、定时任务、事件触发任务和执行参数 |
| Rule 规则 | SOP、审批规则、定价策略、客户分级标准 |
| Event 事件 | 所有业务动作、状态变化、审批、执行结果和异常 |

### 3.4 关系建模与推理链

直接关系负责把企业对象连起来：

```text
Customer 拥有 Contact
Customer 产生 Lead / Order / Contract
Channel 产生 Campaign
Campaign 产生 Lead
Lead 转化为 Customer / Order
Product 包含 SKU / BOM
BOM 依赖 Supplier / Process
Order 关联 Product / Project / Contract / Receivable
Order / Contract 产生 Financial 指标
Project 产生 Task
Task 由 Employee 或 DigitalEmployee 执行
Event 记录所有对象的状态变化
Rule 约束 Task / Approval / Pricing / CustomerTier / Routine
```

本体论真正的价值体现在多跳推理，而不是保存这些关系本身。

制造业推理示例：

```text
Supplier S3 的铝材价格上涨 15%
  → BOM B12 包含 S3 供应的铝材
  → Product P8 的单位成本上涨 7%
  → Customer A 最近询价的订单包含 Product P8
  → 当前报价模板仍按旧成本生成，预计毛利低于规则阈值
  → 战略参谋触发 margin.risk 预警
  → 运营专家生成供应商替代方案
  → 客户专家更新 Customer A 的报价建议
```

这个链路说明本体论不是普通关系数据库，而是让数字员工能从一个局部事件推导出经营影响，并主动触发跨角色协作。

### 3.5 OrgBrain 的三层结构

| 层 | 内容 | 来源 | 用途 |
|----|------|------|------|
| 业务数据层 | 客户、订单、项目、合同、供应商、财务、员工等结构化数据 | ERP / CRM / 财务 / IM / 表格 / 手工录入 | 让数字员工理解当前业务状态 |
| 经验沉淀层 | 项目复盘、报价经验、客户偏好、成功案例、失败教训 | 数字员工执行记录 + 人工补充 | 支撑“上次类似情况怎么做” |
| 规则引擎层 | SOP、审批规则、定价策略、客户分级标准、风险规则 | 管理者设定 + AI 从经验中提炼 | 约束数字员工执行和决策建议 |

### 3.6 events 事件流水

events 是 PilotOne 自迭代的关键底座。没有事件流水，系统只能保存结果，无法理解过程。

每个关键动作都应记录为事件：

- 谁触发了动作。
- 哪个数字员工执行。
- 涉及哪些对象。
- 执行前状态是什么。
- 执行后状态是什么。
- 是否经过审批。
- 产生了什么交付物。
- 结果是否成功。

示例：

```text
event_type: quote.updated
actor: digital_employee.sales
object: quote.Q001
related_customer: customer.A
before: price=10000,status=draft
after: price=9500,status=sent
approval: approved_by=owner
reason: "客户要求 5% 优惠，符合历史折扣规则"
```

### 3.7 技术原则

产品方案阶段不锁定具体数据库。PilotOne 的本体层需要满足的是建模和运行要求，而不是先绑定某个存储实现。

| 能力要求 | 说明 |
|----------|------|
| 结构化查询 | 能稳定查询客户、订单、产品、合同、供应商、渠道等业务对象 |
| 关系遍历 | 能跨对象做多跳关联，例如供应商 → BOM → 产品 → 订单 → 客户 |
| 事件流 | 能记录状态变化、审批、执行结果、误判和异常，为自迭代提供材料 |
| 规则推理 | 能基于客户等级、账期、毛利、供应商风险等规则生成判断和预警 |
| 全文/向量检索 | 能召回合同条款、客户沟通、案例、复盘和非结构化知识 |
| 权限与审计 | 能限制数字员工可见对象和可写操作，所有变更可追踪、可回滚 |

底层实现可以是关系型数据库、图数据库、搜索/向量存储和规则引擎的组合。第一阶段应优先选择工程成本低、可迁移、可观测的混合方案，而不是为了概念完整性过早引入重型知识图谱平台。

原则：

- 能直连现有系统时，优先通过 Bridge 读取和写回。
- 系统没有 API 或需要统一口径时，再同步到 PilotOne 本体层。
- 不追求一次性纳管所有数据，先围绕一个高频场景建最小可用本体。
- 所有 AI 写操作必须有权限、审批和审计记录。

## 4. 产品边界

PilotOne 不是：

- 单纯的 OA、CRM、ERP 或项目管理软件。
- 只会回答问题的企业知识库。
- 一次性定制的 RPA 自动化项目。
- 替代所有员工的黑盒 AI 系统。

PilotOne 是：

- 行业本体和企业实例本体驱动的业务理解层。
- PilotOne-Agent 驱动的数字员工执行层。
- OrgBrain 驱动的组织记忆和自迭代层。
- Bridge 驱动的渐进式系统集成层。

---

## 5. 与 FocusPilot 的关系

FocusPilot 和 PilotOne 共享底层 Agent Engine 思路，但面向不同问题：

- FocusPilot 帮个人建立自迭代执行系统。
- PilotOne 帮企业建立自迭代经营系统。

两者可以共享技术资产，但不应在产品心智上互相包含。FocusPilot 是个人 AIOS，PilotOne 是企业 AI 化操作系统。
