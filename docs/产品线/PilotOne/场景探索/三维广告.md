**宏观趋势**（2024-2026）：
- 中国广告市场整体从流量驱动转向 AI + 内容生态驱动，增速回归 5%-7%
- 广告投放平均成本同比增长 27%，有效转化率仅 16%
- 品牌方自主化趋势加强，广告公司管理的支出占比降至 34.8%
- "All in AI" 成为行业共识，AIGC 技术重塑生产力

**县城广告行业特有困境**：
1. **政府财政紧缩** — 地方财政收入下滑，政府采购预算持续压缩，项目周期拉长
2. **回款周期极长** — 政府项目普遍 6-18 个月回款，部分项目长期挂账
3. **竞争加剧** — 低门槛行业，价格战严重，利润率被压缩
4. **人才外流** — 四五线城市难以留住设计、技术人才
5. **数字化滞后** — 缺乏数据驱动的经营决策能力，依赖经验和关系

## 二、核心困境与痛点

徐小红在沟通中明确指出公司面临三大核心困境：

### 2.1 账款回收困难

- 80% 业务来自政府部门，政府财政收入下滑导致回款周期大幅拉长
- 于都县财政总收入已同比下降，公共预算支出靠转移支付维持，支付能力持续恶化
- 大量应收账款长期挂账，现金流紧张，影响日常运营和新项目承接

### 2.2 必须压缩成本 / 裁员

- 收入端缩水但固定成本（人员、场地、设备）不变，利润率急剧下降
- 需要通过裁员降低人力成本，但同时不能让业务产能崩塌
- 核心矛盾：**人少了，活还得干**

### 2.3 业务转型与扩充

- 不能继续 80% 依赖政府订单，需要开拓民营/企业客户
- 需要从"接单制作"升级为"营销服务商"，提升客户粘性和客单价
- 需要线上获客能力（目前几乎为零），打破地域限制

## 三、OpenClaw 解决方案

基于上述痛点，通过部署 OpenClaw 实现以下 AI 数字化能力：

### 3.1 降本增效模块（解决"人少活多"）

| 海报/展板/文化墙版式自动生成 | AI 根据客户行业、场景自动生成初稿方案 | 设计效率提升 3-5 倍 |
| 方案文本优化 | 校园文化、企业文化墙方案的逻辑梳理与结构优化 | 减少高级设计师工时 |
| 配色/字体/排版建议 | 基于行业风格库自动推荐 | 新手设计师也能输出合格方案 |

#### 3.1.2 供应商比价与成本管控

| 功能 | 说明 | 预期效果 |

### 3.2 现金流改善模块（解决"回款难"）

## 六、技术约束与替代方案

### 无法直接实现

| 需求 | 原因 | 替代方案 |
|------|------|----------|
| ERP 与送货系统数据打通 | 两系统无 API/数据库接口 | 每日导出 Excel → AI 自动合并计算 |
| 直接操控第三方软件 | AI 无法写入闭源系统 | 处理导出数据，实现结果一致 |

### OpenClaw 部署优势

- **多渠道集成**：通过 WhatsApp/微信/企业微信等渠道直接触达客户（催款、关怀、推送）
- **技能插件系统**：比价、ABC 分析、招标监控等均可封装为独立技能
- **本地部署**：数据不出企业，符合政府项目数据安全要求
- **低成本运维**：运行在个人设备上，无需云服务器费用

## 七、数字员工方案对齐映射

> 基于 `10-企业AI数字员工整体方案.md` 的 8 角色框架，适配于都三维广告公司（小型企业，约 10-30 人）的实际场景。

### 7.1 适配原则

数字员工方案原设计面向 50-200 人中型企业，三维广告属于小型县城企业，需要做以下调整：

| 维度 | 原方案（中型企业） | 适配方案（三维广告） |
|------|-------------------|-------------------|
| 角色数量 | 8 个数字员工 | **精简为 4 个**，覆盖最急迫需求 |
| IM 渠道 | 飞书/钉钉 | **微信/企业微信**（县城企业首选） |
| 生产资料层 | CRM/ERP/HR 系统 API 对接 | **Excel 导入 + 轻量自建**（现有系统无 API） |
| 实施周期 | 12-18 个月 | **3-6 个月**完成核心模块 |
| 人类角色 | CEO + 3-5 领域专家 | **徐小红一人决策** + 1-2 骨干员工配合 |

### 7.2 角色裁剪：8 → 4

原方案 8 个角色与三维广告需求的映射关系：

| 原方案角色 | 是否保留 | 理由 | 三维广告中的对应 |
|-----------|---------|------|----------------|
| ① COO 调度 | **合并到秘书** | 小型企业无需独立调度层，老板直接管 | — |
| ② 秘书汇报 | **✅ 保留** | 日报/催收/数据汇总是核心需求 | → **经营助手** |
| ③ 产品研发 | ❌ 不需要 | 广告公司无软件研发需求 | — |
| ④ 项目管理 | ❌ 暂不需要 | 项目规模小，老板直接管控 | — |
| ⑤ 商务销售 | **✅ 保留** | 报价、客户跟进、招标监控是刚需 | → **商务助手** |
| ⑥ 市场内容 | **✅ 保留** | 公众号/短视频/客户关怀是转型关键 | → **内容助手** |
| ⑦ 人事行政 | **部分合并** | 采购比价功能合并到经营助手 | — |
| ⑧ 财务法务 | **✅ 保留** | 催收/毛利分析/成本管控是止血关键 | → **财务助手** |

### 7.3 四个数字员工定义

#### ① 经营助手（合并 COO + 秘书 + 人事行政部分职能）

| 项目 | 内容 |
|------|------|
| **Agent ID** | `ops` |
| **角色定位** | 老板的全能助理：数据查询、日报周报、供应商比价、员工人效分析 |
| **SOUL.md** | 高效务实、数据驱动、善于用表格和图表呈现信息 |
| **核心 Skill** | 日报/周报生成、供应商比价、员工效能分析、ABC 分析、数据合并 |
| **核心 Tool** | `excel_import`（导入 ERP/送货系统数据）、`report_generate`、`procurement_compare`、`work_log_query` |
| **交互对象** | 徐小红（主要）、财务人员 |
| **审批权限** | 纯查询 + 生成报告，不做外发操作 |
| **定时任务** | 每日 18:00 经营日报、每周五周报、每月 ABC 分析报告 |
| **对应需求** | 3.1.2 供应商比价、3.1.3 员工效能、3.2.2 客户毛利分析、3.3.3 订单数据合并 |

**每日 Routine**：

```yaml
routine:
  - time: "08:00"
    action: "检查昨日新导入的 Excel 数据，生成数据质量摘要"
    tool: excel_import + data_validate
    precheck:
      query: "检查 data_inbox/ 目录是否有新文件"
      condition: "new_files > 0"

  - time: "18:00"
    action: "汇总今日订单/出货/回款情况，生成经营日报"
    tool: excel_query + report_generate
    output: "推送到老板微信"

  - time: "每周一 09:00"
    action: "生成上周供应商比价报告，标记异常高价"
    tool: procurement_compare
    precheck:
      query: "检查是否有新的供应商报价数据"
      condition: "new_quotes > 0"

  - time: "每月 1 日 10:00"
    action: "生成上月 ABC 客户/产品分析 + 员工人效排名"
    tool: excel_query + abc_analysis + staff_efficiency
```

#### ② 财务助手（对应原方案 ⑧ 财务法务，专注催收与成本）

| 项目 | 内容 |
|------|------|
| **Agent ID** | `finance` |
| **角色定位** | 应收账款管理、催收提醒、毛利追踪、成本预警 |
| **SOUL.md** | 严谨精确、关注现金流、催款话术得体（政府单位要稳重、企业要专业） |
| **核心 Skill** | 催收话术生成（按客户类型分版本）、逾期分级、对账单生成、毛利趋势分析 |
| **核心 Tool** | `receivable_query`（应收账款查询）、`bill_generate`（账单生成）、`message_send`（微信/企微推送）、`excel_import` |
| **交互对象** | 徐小红（审批催收）、财务人员（数据导入） |
| **审批权限** | 生成催收文案需老板确认后发送；对账单自动生成 |
| **对应需求** | 3.2.1 智能账单催收、3.2.2 客户毛利分析 |

**每日 Routine**：

```yaml
routine:
  - time: "09:00"
    action: "扫描本周到期的应收账款，按逾期天数分级"
    tool: receivable_query
    precheck:
      query: "检查是否有本周到期或已逾期的应收款"
      condition: "due_count > 0"
    output: |
      【今日应收提醒】
      ■ 本周到期：3 笔，合计 12.5 万
      ■ 逾期 30 天内：5 笔，合计 28 万
      ■ 逾期 90 天+：2 笔，合计 15 万（建议升级催收策略）

  - time: "每周一 10:00"
    action: "生成上周催收执行报告 + 本周催收计划"
    tool: receivable_query + bill_generate
    output: "催收清单推送给老板，附带建议话术"

  - time: "每月 5 日"
    action: "生成上月全量对账单，按客户分别生成"
    tool: bill_generate
    output: "对账单待老板确认后，逐个推送给客户"
```

#### ③ 商务助手（对应原方案 ⑤ 商务销售，专注招标 + 报价）

| 项目 | 内容 |
|------|------|
| **Agent ID** | `biz` |
| **角色定位** | 招投标信息监控、报价方案生成、客户跟进、商机分析 |
| **SOUL.md** | 专业严谨、熟悉政府采购流程、报价准确、商务礼仪到位 |
| **核心 Skill** | 招标信息抓取与筛选、报价模板生成、投标条件匹配分析、客户跟进提醒 |
| **核心 Tool** | `bid_monitor`（招标网站爬虫）、`bid_analyze`（资质匹配分析）、`quote_generate`（报价生成）、`message_send` |
| **交互对象** | 徐小红（审批报价）、业务人员（现场跟进） |
| **审批权限** | 招标信息自动推送；报价方案需老板确认 |
| **对应需求** | 3.3.1 招投标监控、3.3.3 客户画像与商机分析 |

**每日 Routine**：

```yaml
routine:
  - time: "07:30"
    action: "抓取赣州及周边招标网站新公告，按关键词筛选"
    tool: bid_monitor
    keywords: [广告制作, 文化墙, 目视化, 校园文化, 展馆, 标识标牌, 宣传栏]
    scope: [于都县, 赣州市, 江西省]
    precheck:
      query: "抓取最近 24 小时新发布的招标公告"
      condition: "new_bids > 0"
    output: |
      【今日招标速报】
      ■ 新增 3 条匹配公告：
      1. 于都县XX小学校园文化建设项目 | 预算 45 万 | 截止 4/15
         → 资质匹配度：✅ 符合（需营业执照+3个类似案例）
      2. 赣州市XX医院导视系统采购 | 预算 22 万 | 截止 4/12
         → 资质匹配度：⚠️ 需确认（要求设计资质）

  - time: "14:00"
    action: "检查近 30 天未联系的老客户，生成跟进建议"
    tool: customer_query(last_contact_days>30)
    precheck:
      query: "检查沉默客户数量"
      condition: "silent_count > 0"
```

#### ④ 内容助手（对应原方案 ⑥ 市场内容，专注获客）

| 项目 | 内容 |
|------|------|
| **Agent ID** | `content` |
| **角色定位** | 公众号文章、短视频脚本、节日问候、个人 IP 打造 |
| **SOUL.md** | 创意导向、熟悉各平台调性、文案接地气、懂广告行业专业术语 |
| **核心 Skill** | 种草文章写作（设计思路+材质工艺+落地效果）、短视频脚本、节日文案（按客户行业分版本）、SEO 关键词优化 |
| **核心 Tool** | `content_generate`（文案生成）、`message_send`（多渠道发送）、`case_library_query`（公司案例库查询） |
| **交互对象** | 徐小红（审批内容）、设计师（提供素材） |
| **审批权限** | 内容生成后需老板确认再发布 |
| **对应需求** | 3.3.2 线上获客与品牌建设 |

**每日 Routine**：

```yaml
routine:
  - time: "08:00"
    action: "生成今日公众号种草文章（基于案例库轮转）"
    tool: case_library_query + content_generate
    output: "文章草稿推送给老板审核，含标题/正文/配图建议"

  - time: "10:00"
    action: "生成今日短视频脚本（抖音/小红书/视频号各一版）"
    tool: content_generate
    output: "脚本草稿含：口播文案、字幕、标题、话题标签"

  - time: "节假日前 3 天（事件触发）"
    action: "生成节日问候文案，按客户行业分版本"
    tool: content_generate + customer_query
    versions:
      - 政府单位版（稳重正式）
      - 企业商务版（专业大方）
      - 商家亲切版（温馨活泼）
    output: "文案列表待老板确认后定时发送"
```

### 7.4 生产资料层适配

三维广告没有 CRM/API，需要用轻量方案替代：

| 原方案生产资料 | 三维广告替代方案 | 说明 |
|--------------|----------------|------|
| CRM 系统 | **Excel/WPS 客户台账** + 微信聊天记录 | 老板手动维护或 AI 辅助整理 |
| ERP API | **每日导出 Excel → AI 导入** | 方式 4（文件读取）最现实 |
| 送货系统 API | **每日导出 Excel → AI 导入** | 同上 |
| 财务系统 | **Excel 台账** | 县城小企业大概率没有用友/金蝶 |
| 项目管理/看板 | **不需要** | 项目少，老板直接管 |
| HR 系统 | **不需要** | 人少，不需要系统 |
| 知识库（向量记忆） | **公司案例库（QMD）** | 将历史项目照片、方案文档向量化 |

**数据接入策略**（对应原方案 4.5 节）：

```
三维广告的现实路径：

优先级   方式                 适用系统                 备注
──────────────────────────────────────────────────────
1️⃣      Excel 文件导入        ERP、送货、财务台账       每日手动或定时导出
2️⃣      微信消息接入          客户沟通、催收回复         OpenClaw 微信渠道
3️⃣      网页爬虫              招标网站                  bid_monitor Tool
4️⃣      案例库向量化          历史项目文档/照片          QMD 一次性导入

不需要：数据库直连、API 逆向、RPA（现有系统太简单）
```

### 7.6 实施路径映射

将原方案的 4 阶段（Phase 0-3）与三维广告的三期（止血-降本-开源）对齐：

| 原方案阶段 | 三维广告阶段 | 周期 | 上线角色 | 核心交付 |
|-----------|------------|------|---------|---------|
| Phase 0 基础设施 | **第一期前置** | 1 周 | — | 部署 OpenClaw + 微信接入 + Excel 导入 Tool |
| Phase 1 秘书+COO | **第一期：止血** | 2 周 | 经营助手 + 财务助手 | 催收自动化 + ABC 分析 + 人效分析 |
| Phase 2 商务+研发 | **第二期：降本** | 2-4 周 | 商务助手 | 招标监控 + 报价生成 + 供应商比价 |
| Phase 3 全职能 | **第三期：开源** | 1-3 月 | 内容助手 | 公众号日更 + 短视频 + 客户关怀 |
| Phase 4 优化自治 | **持续迭代** | 3 月+ | 全部 | 自迭代 SOP + 审批规则放松 |

### 7.7 关键差异与简化点

| 原方案机制 | 三维广告是否需要 | 说明 |
|-----------|:---------------:|------|
| LangGraph 工作流引擎 | ❌ 暂不需要 | 业务流程简单，无跨多天多步流程 |
| 跨角色协作（COO 调度） | ❌ 不需要 | 老板直接与每个助手对话 |
| 权限隔离矩阵 | ⚠️ 极简版 | 4 个员工各管各的，老板全权限 |
| 事件流水表 | ✅ 需要 | 用于生成日报和追踪催收记录 |
| 两阶段 Routine | ✅ 需要 | 避免无变化时骚扰老板 |
| 自迭代闭环 | ⚠️ Phase 2+ | 先跑起来，积累数据后再开启 |
| 数据时间衰减 | ❌ 暂不需要 | 数据量小，不存在冷热分区问题 |
| 审批规则引擎 | ⚠️ 极简版 | 查询类自动执行，对外发送一律审批 |
