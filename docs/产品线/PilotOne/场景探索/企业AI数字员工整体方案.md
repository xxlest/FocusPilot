# 企业 AI 数字员工整体方案

> 基于 OpenClaw 架构的中型企业（50-200人）全职能 AI 驱动运营方案

## 一、方案概述

### 1.1 目标

构建一个 **AI 驱动的企业运营体系**，覆盖产品、研发、项目管理、销售、市场、客服、HR、财务、行政法务、高管决策共 10 个职能域。人类仅负责**战略决策、专业判断和物理在场**，其余执行工作全部由 AI 数字员工完成。

### 1.2 核心设计理念

| 原则 | 说明 |
|------|------|
| **Agent 无状态，流程有状态** | Agent 本身不依赖常驻进程，但跨多轮的业务流程通过工作流状态引擎（LangGraph）持久化跟踪 |
| **Profile 定义角色** | 8 套 Profile 配置包 = 8 个数字员工，无需 8 个独立系统 |
| **生产资料外置** | 业务数据通过 Tool/Skill 按需读写，不存储在 Agent 内部 |
| **渐进式披露** | Agent 只加载当前任务所需的上下文，不一次性灌入全部数据 |
| **人类只做审批** | 数字员工先出方案/草稿，人类确认后再执行，日常小事可自主 |

### 1.3 约束条件

- 统一通过企业 IM（飞书/钉钉/企业微信）交互
- 先设计理想终态架构，分阶段实施
- 中型企业（50-200人），部门齐全

## 二、架构设计

### 2.1 整体分层架构

```
┌───────────────────────────────────────────────────────┐
│                    人类交互层                           │
│  决策者（CEO + 3-5领域专家）  现场执行者（若干）          │
├───────────────────────────────────────────────────────┤
│                    渠道接入层                           │
│  企业 IM（飞书/钉钉）统一接入                           │
│  ← OpenClaw 渠道适配能力直接复用                        │
├───────────────────────────────────────────────────────┤
│                    调度层                               │
│  路由引擎：根据消息来源/内容 → 匹配 Profile → 注入上下文  │
│  审批引擎：低风险自主执行，高风险等待人类确认              │
│  ← LangGraph 有状态工作流                               │
├───────────────────────────────────────────────────────┤
│                    Agent 执行层                         │
│  通用 Agent 引擎 + 8 套 Profile 配置包                  │
│  ← OpenClaw 无状态 Agent + CrewAI 多角色协作             │
├───────────────────────────────────────────────────────┤
│                    能力层                               │
│  Tool（插件）：读写外部业务系统的"手脚"                   │
│  Skill（技能）：教 Agent 怎么组合工具的"教程"             │
│  ← OpenClaw Plugin SDK                                 │
├───────────────────────────────────────────────────────┤
│                    生产资料层（需自建）                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │ 项目管理  │ │ CRM/ERP  │ │ 知识库   │ │ 工作日志  │ │
│  │ 任务看板  │ │ 财务系统  │ │ 文档库   │ │ 日报周报  │ │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘ │
│  ← 对接企业已有系统 或 自建轻量存储                      │
└───────────────────────────────────────────────────────┘
```

### 2.2 单次调用的执行流程

```
触发事件（IM 消息 / 定时任务 / 审批回调）
    │
    ▼
路由层：识别来源 → 匹配 Profile
    │
    ▼
组装上下文（自动完成，无需手动配置 System Prompt）：
  ├─ ① SOUL.md        — 角色人格（必带）
  ├─ ② AGENTS.md      — 行为规则 + 公司背景（必带）
  ├─ ③ IDENTITY.md    — 身份信息（必带）
  ├─ ④ USER.md        — 用户偏好（必带）
  ├─ ⑤ 可用工具列表    — 含插件注册的 Tool（自动注入）
  ├─ ⑥ 可用技能列表    — Skill 按需加载详情（自动注入）
  ├─ ⑦ 对话历史        — 按 Session Key 隔离（动态加载）
  └─ ⑧ 向量记忆        — 语义搜索相关片段（动态加载）
    │
    ▼
临时 Agent 执行
  ├─ 过程中按需调用 Tool（查 CRM、写看板、查组织架构...）
  └─ Skill 指导操作流程和工具组合顺序
    │
    ▼
结果处理
  ├─ 写回生产资料层（更新任务状态、记录工作日志...）
  ├─ 回复渠道（IM 消息返回给人类）
  └─ Agent 销毁
```

### 2.3 Session Key 隔离策略

延用 OpenClaw 的 Session Key 机制，按 `agent:{agentId}:{channel}:{chatType}:{peerId}` 隔离对话上下文：

| 场景 | Session Key 示例 | 说明 |
|------|-----------------|------|
| CEO 私聊 COO 员工 | `agent:coo:feishu:dm:ceo-id` | CEO 与 COO 的独立对话历史 |
| 商务专家在销售群 | `agent:biz:feishu:group:sales-group` | 销售群共享上下文 |
| 定时生成日报 | `agent:secretary:cron:daily-report` | 定时任务独立 session |

## 四、生产资料体系

### 4.1 生产资料的本质

**生产资料 = 持久化的业务数据 + 读写接口**。Agent 无状态，所有业务状态都外置在生产资料层。

### 4.2 三种承载方式

| 生产资料类型 | 承载方式 | 在 OpenClaw 中的实现 | 加载时机 |
|-------------|---------|---------------------|---------|
| 公司信息、组织架构、规章制度 | **Bootstrap File**（AGENTS.md / SOUL.md） | 工作区静态文件，每次必带 | 每次调用自动注入 |
| CRM、项目看板、财务系统、工作日志 | **Tool（插件）** | Plugin SDK 注册自定义工具 | Agent 执行中按需调用 |
| 操作规范、流程指导 | **Skill（技能文件）** | SKILL.md 文件，按需加载 | 任务匹配时加载 |
| 历史文档、客户资料、知识库 | **向量记忆（QMD）** | QMD 向量化 + 语义搜索 | 每次调用自动搜索相关片段 |

### 4.3 渐进式披露机制

不是把所有数据一次性灌入 Agent 上下文，而是分层按需加载：

```
第一层（每次必带，~2000 token）：
  ├─ Profile（角色定义）
  ├─ 公司背景（AGENTS.md）
  └─ 当前消息上下文

第二层（自动搜索，~1000 token）：
  ├─ 向量记忆搜索结果（相关历史）
  └─ 对话历史（最近 N 轮）

第三层（Agent 主动调用，按需）：
  ├─ 查 CRM → crm_query Tool
  ├─ 查项目看板 → task_board_query Tool
  ├─ 查财务数据 → finance_system_query Tool
  └─ ...任何需要的数据

第四层（执行后写回）：
  ├─ 更新任务状态 → task_board_write Tool
  ├─ 记录工作日志 → work_log_write Tool
  └─ 更新 CRM → crm_write Tool
```

### 4.4 Tool 插件实现示例

以 OpenClaw Plugin SDK 为基础，每个业务系统封装为一个插件：

```typescript
// extensions/enterprise-data/index.ts
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

export default definePluginEntry({
  id: "enterprise-data",
  name: "企业生产资料插件",
  description: "对接企业业务系统的统一数据读写层",

  register(api) {
    // ━━━ 项目管理 ━━━
    api.registerTool({
      name: "task_board_query",
      description: "查询项目任务看板。支持按项目名、负责人、状态、时间范围筛选。",
      parameters: TaskQuerySchema,
      async execute(_id, params) {
        return await fetchFromProjectSystem("/tasks", params);
      },
    });

    api.registerTool({
      name: "task_board_write",
      description: "创建或更新任务。可设置标题、描述、负责人、截止日期、状态、优先级。",
      parameters: TaskWriteSchema,
      async execute(_id, params) {
        return await postToProjectSystem("/tasks", params);
      },
    });

    // ━━━ CRM ━━━
    api.registerTool({
      name: "crm_query",
      description: "查询客户信息、商机、历史报价、合同状态。",
      parameters: CrmQuerySchema,
      async execute(_id, params) {
        return await fetchFromCRM("/query", params);
      },
    });

    api.registerTool({
      name: "crm_write",
      description: "更新客户信息、创建商机、记录跟进。需要审批的操作会返回待审批状态。",
      parameters: CrmWriteSchema,
      async execute(_id, params) {
        return await postToCRM("/update", params);
      },
    });

    // ━━━ 工作日志 ━━━
    api.registerTool({
      name: "work_log_write",
      description: "记录工作内容到工作日志。包含：执行人(数字员工ID)、所属项目、工作描述、耗时、关联任务ID。",
      parameters: WorkLogWriteSchema,
      async execute(_id, params) {
        return await appendToWorkLog(params);
      },
    });

    api.registerTool({
      name: "work_log_query",
      description: "查询工作日志。支持按日期、员工、项目、关键词筛选。用于生成日报周报。",
      parameters: WorkLogQuerySchema,
      async execute(_id, params) {
        return await queryWorkLog(params);
      },
    });

    // ━━━ 组织架构 ━━━
    api.registerTool({
      name: "org_query",
      description: "查询公司组织架构、人员信息、汇报关系、部门结构。",
      parameters: OrgQuerySchema,
      async execute(_id, params) {
        return await fetchFromHRSystem("/org", params);
      },
    });

    // ━━━ 财务系统 ━━━
    api.registerTool({
      name: "finance_query",
      description: "查询财务数据：营收、成本、利润、应收应付、预算执行率。",
      parameters: FinanceQuerySchema,
      async execute(_id, params) {
        return await fetchFromFinanceSystem("/query", params);
      },
    });
  },
});
```

### 4.5 企业系统数据接入策略

现实中企业系统参差不齐，并非所有系统都有开放 API。按可行性排序的四种接入方式：

```
优先级   方式              适用场景                  开发成本   数据质量
───────────────────────────────────────────────────────────────────
1️⃣      官方 API          飞书/钉钉/Salesforce 等    低         高
2️⃣      数据库只读直连     用友/金蝶等有数据库的系统   中         高
3️⃣      API 逆向探测       有 Web 界面的内部系统      中         中
4️⃣      RPA 屏幕操作       完全封闭的桌面软件         高         低
```

#### 方式 1：官方 API（优先使用）

现代 SaaS 和企业 IM 都提供 REST/GraphQL API，直接对接：

| 系统 | API 质量 | 说明 |
|------|---------|------|
| 飞书/钉钉/企业微信 | 优 | 开放平台，文档齐全 |
| Salesforce/HubSpot | 优 | 标准 REST API |
| Jira/Confluence | 良 | REST API 完善 |
| GitHub/GitLab | 优 | GraphQL + REST |

#### 方式 2：数据库只读直连（国内 ERP/财务系统的现实方案）

国内大量企业使用用友、金蝶、浪潮等系统，这些系统的 API 通常不完善或不开放，但底层数据库（SQL Server/Oracle/MySQL）可以直连：

```
操作步骤：
1. 找 DBA 申请只读数据库账号
2. 分析表结构，找到业务实体对应的表
3. 建只读视图（View），封装复杂的多表关联
4. Tool 通过 SQL 查询视图获取数据

示例：
  用友 U8 → SQL Server
  → 创建视图 v_customers, v_orders, v_invoices
  → Tool 执行 SELECT * FROM v_customers WHERE name = '客户A'

注意：写操作不能走数据库直连（会绕过业务校验），
     写操作应通过官方界面或 API 探测方式完成
```

#### 方式 3：API 逆向探测（有 Web 界面的系统）

企业内部 Web 系统自身也是前端调后端 API，通过浏览器抓包可以发现并封装这些接口：

```
步骤：
1. 打开系统 Web 界面，F12 → Network 面板
2. 操作一遍目标功能（查订单、提审批...）
3. 记录抓到的 HTTP 请求（URL、Headers、参数格式）
4. 封装为 OpenClaw Tool

示例：
  抓包发现 OA 审批系统接口：
  GET /api/v1/approvals?status=pending
  Headers: { X-Token: "xxx" }

  封装为 Tool：
  api.registerTool({
    name: "oa_query",
    async execute(_id, params) {
      return await fetch(OA_URL + "/api/v1/approvals", {
        headers: { "X-Token": await refreshToken() }
      });
    }
  });

风险与应对：
  · Session/Token 过期 → 定时自动登录刷新
  · 接口变更（厂商升级）→ 加监控告警，变更时修复
  · 被限频/反爬 → 控制调用频率，走内网
  · 写操作风险大 → 探测的 API 建议只做读取
```

#### 方式 4：RPA 屏幕操作（最后手段）

针对完全封闭的 C/S 架构桌面软件，使用 RPA 模拟人类操作：

```
Playwright / Puppeteer / 按键精灵
  → 模拟鼠标键盘操作 ERP 界面
  → 截图 + OCR 提取数据
  → 封装成 Tool

缺点：慢、不稳定、维护成本高
仅用于极少数完全封闭的系统
```

### 4.6 事件流水表（轻量时间层）

所有 Tool 的写操作自动记录一条事件日志，用于追踪业务对象的状态变化轨迹：

```sql
-- 一张表记录所有业务实体的状态变更
CREATE TABLE events (
  id          BIGSERIAL PRIMARY KEY,
  entity_type VARCHAR(50)  NOT NULL,  -- "customer", "project", "contract"
  entity_id   VARCHAR(100) NOT NULL,  -- 业务实体 ID
  action      VARCHAR(50)  NOT NULL,  -- "status_change", "created", "updated"
  old_value   TEXT,                    -- 变更前（可选）
  new_value   TEXT,                    -- 变更后
  actor       VARCHAR(50)  NOT NULL,  -- 执行的数字员工 ID（"biz", "pm"...）
  context     TEXT,                    -- 简要描述
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);

-- 索引：按实体查轨迹、按时间查日报
CREATE INDEX idx_events_entity ON events(entity_type, entity_id, created_at);
CREATE INDEX idx_events_time ON events(created_at);
CREATE INDEX idx_events_actor ON events(actor, created_at);
```

Tool 写操作时顺带插入：

```typescript
// 在每个写操作 Tool 的 execute 末尾加一行
async execute(_id, params) {
  const result = await postToCRM("/update", params);
  // 记录事件
  await db.query(
    `INSERT INTO events (entity_type, entity_id, action, new_value, actor, context)
     VALUES ($1, $2, $3, $4, $5, $6)`,
    ["customer", params.id, "updated", JSON.stringify(params), "biz", "更新客户信息"]
  );
  return result;
}
```

用途：
- 秘书员工生成日报：`SELECT * FROM events WHERE created_at >= today()`
- 查看客户轨迹：`SELECT * FROM events WHERE entity_type='customer' AND entity_id='c-001' ORDER BY created_at`
- 审计追踪：谁在什么时候通过哪个数字员工做了什么操作

#### Routine 在 OpenClaw 中的实现

利用 OpenClaw 已有的 heartbeat 机制，每个 Agent 配置自己的 routine：

```yaml
# openclaw.yaml 中的 Agent 配置
agents:
  list:
    - id: biz
      identity: { name: "商务小王" }
      heartbeat:
        enabled: true
        schedule:
          - cron: "0 8 * * 1-5"    # 工作日 08:00
            message: "执行早间 routine：检查新线索"
          - cron: "0 10 * * 1-5"   # 工作日 10:00
            message: "执行上午 routine：跟进昨日报价"
          - cron: "0 14 * * 1-5"   # 工作日 14:00
            message: "执行下午 routine：扫描沉默客户"
          - cron: "0 17 * * 1-5"   # 工作日 17:00
            message: "执行收尾 routine：更新 CRM，提交工作摘要"
```

heartbeat 触发时，Agent 被临时创建，加载 Profile + Skill，执行 routine 定义的动作，写入结果，然后销毁——跟响应用户消息的流程一致，只是触发方式从"用户消息"变成了"定时心跳"。

#### Routine 的两阶段执行（成本优化 + 噪音控制）

直接为每次 routine 创建完整 Agent（加载 Profile → 调 LLM → 生成回复）成本高、噪音大。实际采用**两阶段模式**：

```
阶段 1：轻量检测（不调 LLM，纯 SQL/API 查询）
  → 检查是否有变化：新线索？新报价回复？超时任务？
  → 与上次 routine 结果对比
  → 无变化 → 跳过，不触发 Agent，不推送任何消息
  → 有变化 → 进入阶段 2

阶段 2：完整执行（创建 Agent，调 LLM）
  → 加载 Profile + Skill
  → 分析变化内容，生成有价值的输出
  → 推送给相关人员
```

实现方式——在 heartbeat 配置中加 `precheck`（轻量检测脚本）：

```yaml
agents:
  list:
    - id: biz
      heartbeat:
        schedule:
          - cron: "0 10 * * 1-5"
            message: "跟进昨日报价"
            precheck:
              # 阶段1：纯 SQL 检查，不调 LLM
              query: "SELECT count(*) FROM quotes WHERE status='pending' AND age > interval '48h'"
              condition: "result > 0"
              # 只有存在超过 48h 未回复的报价时，才创建 Agent 执行
```

**效果**：
- 成本降低 70-80%（大部分 routine 在阶段 1 就结束了）
- 噪音消除（"今日无新事项"不再推送）
- 真正有变化时，推送的每条消息都有价值

#### Routine 输出的变化检测

即使进入阶段 2，推送结果也需要与上次对比，避免重复推送相同内容：

```
Day 1: "发现 3 个沉默客户：客户A、B、C" → 推送 ✅
Day 2: 阶段1检测 → 还是那 3 个 → 跳过 ❌（不推送）
Day 5: 阶段1检测 → 客户D 新增 → 进入阶段2
        "新增 1 个沉默客户：客户D（客户A/B/C 仍未跟进）" → 推送 ✅
```

实现：每次 routine 执行后，将结果摘要（如实体 ID 列表）写入 `routine_last_result` 表，下次执行时先对比。

### 5.3 自迭代闭环

每个数字员工内置**周复盘 routine**，自动发现工作模式中的改进点：

```
┌─────────────────────────────────────────────────┐
│               自迭代闭环                          │
│                                                  │
│  ① 执行 SOP（每日 routine + 响应请求）            │
│        ↓                                         │
│  ② 记录结果（events 表 + 工作日志）               │
│        ↓                                         │
│  ③ 周复盘（每周五自动触发）                        │
│     · 扫描本周 events，统计执行模式                │
│     · 发现重复问题 / 效率瓶颈 / 异常偏差           │
│     · 生成改进建议（每员工每月最多 3 条，只挑最重要的）│
│        ↓                                         │
│  ④ 人类审批                                      │
│     · 改进建议推送给对应领域专家                    │
│     · 专家批准 / 拒绝 / 修改                      │
│        ↓                                         │
│  ⑤ 更新 SOP（Skill 文件 / 审批规则 / Routine）    │
│        ↓                                         │
│  ① 下周按新 SOP 执行（循环）                      │
└─────────────────────────────────────────────────┘
```

#### 场景举例：商务员工的自迭代

```
Week 1-4 执行记录：
  events 表数据：
  · 生成报价 12 份
  · 其中 9 份被客户要求降价（命中率 75%）
  · 降价幅度：5% x 6 份，8% x 2 份，10% x 1 份
  · 报价到签约平均周期：4.2 天

Week 4 周复盘 routine 自动分析：
  发现 1："75% 的报价被要求降价 5-10%"
  → 建议："将标准产品默认报价下调 5%"
  → 预期效果："减少 60% 的报价来回修改，签约周期缩短至 2 天"

  发现 2："周三下午的报价响应时间平均 4 小时，其他时段 1 小时"
  → 建议："周三下午可能有周期性会议冲突，调整 routine 时间"

  发现 3："客户 B 行业的报价全部被拒"
  → 建议："该行业可能需要差异化定价策略，建议商务专家评估"

推送给商务专家：
  专家："发现1同意执行；发现2不需要调整；发现3很有价值，我来制定策略"

更新 Skill 文件：
  报价规则新增："标准产品初始报价在基准价基础上预留 5% 谈判空间"
```

#### 场景举例：秘书员工的自迭代

```
Week 1-4 执行记录：
  · 生成日报 20 份
  · CEO 对日报的反馈：3 次要求补充项目风险信息
  · CEO 从未查看"人事行政"板块

Week 4 周复盘：
  发现 1："CEO 多次要求补充风险信息"
  → 建议："日报模板增加'风险提示'板块，自动从 events 中提取延期/超预算事件"

  发现 2："人事行政板块从未被查看"
  → 建议："将人事行政信息从日报移至周报，日报只保留核心经营数据"

审批后更新日报模板（Skill 文件）
→ 下周日报自动包含风险提示，不再包含人事行政琐事
```

#### 场景举例：产品研发员工的自迭代

```
Month 1-2 执行记录：
  · Code Review 87 次
  · 其中 23 次指出"缺少错误处理"（占 26%）
  · 涉及 5 位开发者，其中 2 位占 18 次

Month 2 月度复盘：
  发现："26% 的 Code Review 发现缺少错误处理，集中在 2 位开发者"
  → 建议 1："在 CI 中增加错误处理覆盖率检查"
  → 建议 2："为这 2 位开发者生成针对性的错误处理最佳实践文档"

  更新 Skill："Code Review 时优先检查新提交是否包含错误处理"
  产出交付物："错误处理最佳实践指南"推送给相关开发者
```

#### 自迭代的防疲劳机制

防止改进建议泛滥导致审批者疲劳：

| 机制 | 规则 | 原因 |
|------|------|------|
| **数量限制** | 每个员工每月最多输出 3 条改进建议 | 逼 Agent 只挑最重要的，专家每月最多审批 24 条（8×3） |
| **去重检测** | 与历史建议对比，相同建议不重复提出 | 被拒绝的建议记录拒绝原因，避免反复建议 |
| **累积触发** | 同一模式连续 3 周出现才生成建议 | 过滤掉偶发现象，只报告持续性问题 |
| **Skill 变更验证** | 更新 Skill 后用 2-3 个典型场景做冒烟测试 | 防止新规则与旧规则冲突或产生反效果（Phase 2+ 实施） |

### 5.4 产物的时间衰减与归档策略

所有产物都需要时间维度管理，防止累积资产变成"数据垃圾场"。

#### 交付物：按业务状态归档

交付物不按时间一刀切，而是按**业务状态**驱动归档：

```
报价单：
  · 已签约    → 归档，关联到合同记录
  · 已失效    → 归档，标记失败原因（价格？竞品？需求变更？）
  · 3 月未响应 → 自动标记"过期"，提醒商务跟进或归档

合同草案：
  · 已签署    → 归档到合同管理，开始跟踪履约
  · 被拒绝    → 归档，记录拒绝原因
  · 6 月未签署 → 标记"可能流失"，触发跟进

PRD / 技术方案：
  · 已上线    → 归档到项目文档库
  · 已废弃    → 归档，标记废弃原因（需求变更？技术不可行？）

日报 / 周报：
  · 近 1 月   → 全文保留
  · 1-6 月    → 保留摘要和关键指标
  · 6 月+     → 只保留月度聚合数据
```

#### 累积资产：分级衰减

```
资产类型          热区(0-3月)   温区(3-12月)   冷区(12月+)
───────────────────────────────────────────────────────────
客户画像          全量保留       全量保留        全量保留
                                              （但标记 >6月未
                                               验证的联系人为
                                              "待确认"）

事件流水          全量保留       保留摘要        压缩归档
(events 表)       （热查询）     删除详情字段     只保留统计值
                               old/new_value   和关键事件

知识库 / FAQ      全量保留       全量保留        按引用频率衰减
                                              · 近6月被引用 → 保留
                                              · 6-12月未引用 → 标记
                                                "待审核"
                                              · 12月+ 未引用 → 归档

工作日志          逐条保留       按周压缩为摘要   按月压缩为摘要
```

#### 自迭代产物：强衰减

```
SOP 改进建议：
  · 已采纳    → 合并进 Skill 文件，建议本身归档
  · 30 天未审批 → 自动关闭，标记"过期未处理"
  · 被拒绝    → 记录拒绝原因，归档（避免重复建议）

效率指标：
  · 近 6 月   → 保留日粒度趋势数据
  · 6-12 月   → 压缩为周粒度
  · 12 月+    → 只保留月度均值
  · 年度快照  → 永久保留（用于同比分析）

Skill 文件 (SOP 本身)：
  · 不衰减，不删除
  · 用 Git 版本控制，每次修改留历史
  · 可回滚到任意版本
```

#### 数据治理 Routine（秘书员工每月执行）

```yaml
# 秘书员工的月度数据治理 routine
- time: "每月最后一个工作日 15:00"
  action: "执行数据治理扫描"
  steps:
    - 扫描过期交付物（>3月未响应的报价、>6月未签的合同）
      → 标记过期，生成跟进或归档建议
    - 压缩温区事件流水（3-12月，删除详情字段）
    - 扫描未引用知识条目（>6月未被 Agent 引用）
      → 标记"待审核"，推送给相关专家确认是否保留
    - 压缩老工作日志（>1月按周聚合，>6月按月聚合）
    - 关闭超期未审批的 SOP 改进建议
    - 生成"数据治理报告"推送到经营驾驶舱群
  output: |
    【4月数据治理报告】
    ■ 归档：过期报价 3 份，废弃 PRD 1 份
    ■ 压缩：事件流水压缩 12,000 条 → 800 条摘要
    ■ 待审核：知识库 5 条长期未引用条目（已推送专家）
    ■ 关闭：2 条过期未审批的 SOP 改进建议
    ■ 存储节约：本月释放约 15% 存储空间
```

#### 冷区数据的"按需解冻"（Phase 3+ 实施）

归档不等于删除。冷区数据应保持**可访问但慢查询**的状态：

```
CEO："去年客户A的报价策略是什么？"
    ↓
Agent 在热区/温区未找到详情
    ↓
Agent 自动申请解冻冷区数据（查询归档存储）
    ↓
返回历史详情（响应时间 3-5 秒，vs 热区 <1 秒）
```

实现：冷区数据归档到独立的 PostgreSQL 分区表或对象存储（S3/MinIO），Agent 通过专用 Tool `archive_query` 按需查询。

### 5.5 已识别风险的分阶段处理计划

基于对抗性审查，以下问题已识别但安排在后续阶段处理：

| 问题 | 安排阶段 | 处理方式 |
|------|---------|---------|

## 六、人类角色重新定义

### 6.1 核心观点

AI 数字员工体系建立后，企业中大量"信息搬运"岗位被 AI 直接替代。人类的不可替代性来自三个硬约束：**法律主体、专业判断、物理在场**。

### 6.2 人类角色分层

```
┌─────────────────────────────────────────────────────────┐
│  决策层（必须是人，1 人）                                  │
│  CEO / 创始人                                            │
│  职责：战略决策、最终审批、对外代表                          │
│  交互：直接与 AI 对话，问什么答什么，审批关键动作             │
├─────────────────────────────────────────────────────────┤
│  判断层（暂时需要人，3-5 人）                               │
│  领域专家（不是管理者，是专家）                              │
│  ├─ 商务专家：大客户谈判、定价策略、合同条款                  │
│  ├─ 技术专家：架构决策、技术选型、故障定性                    │
│  ├─ 财务/法务专家：税务筹划、合规审查、风险评估               │
│  └─ 人才专家：关键岗位终面、文化判断、组织诊断                │
│  交互：AI 准备材料 → 专家判断 → AI 执行                     │
├─────────────────────────────────────────────────────────┤
│  现场层（物理世界需要人，按业务性质若干）                      │
│  ├─ 销售现场：面对面拜访、展会、客户接待                      │
│  ├─ 交付现场：实施部署、培训、验收                            │
│  └─ 运维现场：硬件维护、物理操作                             │
│  交互：AI 提前准备 → 人执行 → 人回填结果 → AI 后续            │
├─────────────────────────────────────────────────────────┤
│  ❌ 以下角色被 AI 吸收（不再需要专职人类）                    │
│  行政助理、报表分析师、客服一线、项目协调员                    │
│  内容编辑、招聘筛选、财务核算、采购比价                       │
└─────────────────────────────────────────────────────────┘
```

### 6.4 工作流状态引擎（解决"无状态 Agent 跟踪多步流程"的问题）

Agent 本身无状态，但很多业务流程跨多轮、跨多天（如报价流程：Day1 提需求 → Day3 出方案 → Day5 客户确认）。这由 **LangGraph 工作流状态引擎**解决：

```
┌─────────────────────────────────────────────────┐
│  LangGraph 工作流状态引擎（调度层，常驻）          │
│                                                  │
│  每个业务流程 = 一个有状态的 Graph 实例             │
│  ┌──────────────────────────────────┐            │
│  │ 报价流程 #2024-001               │            │
│  │ 当前节点：等待客户确认             │            │
│  │ 关联 Session Key：agent:biz:...  │            │
│  │ 参与角色：商务员工、财务法务员工    │            │
│  │ 历史步骤：[提需求→出方案→改价格]   │            │
│  │ 超时规则：3天未确认 → 自动催办      │            │
│  └──────────────────────────────────┘            │
│                                                  │
│  Agent 每次执行时：                                │
│  1. 通过 Tool 查询当前流程状态                     │
│  2. 执行当前步骤的动作                             │
│  3. 通过 Tool 推进流程到下一个节点                  │
│  → Agent 销毁，流程状态留在 LangGraph              │
└─────────────────────────────────────────────────┘
```

**关键设计**：Agent 通过 `workflow_query` 和 `workflow_advance` 两个 Tool 与状态引擎交互，自身不保存任何流程状态。

### 6.5 跨角色协作机制

当一个任务涉及多个数字员工（如客户要求"合同加 SLA 条款 + 修 bug"），COO 调度员工通过 LangGraph 创建**并行子流程**：

```
客户请求（跨角色）
    │
    ▼
COO 员工：创建协调流程（LangGraph Graph 实例）
    ├─ 子流程 A：商务员工 → 修改合同条款
    ├─ 子流程 B：财务法务员工 → 审查 SLA 法律风险
    ├─ 子流程 C：产品研发员工 → 修复 bug
    └─ 子流程 D：项目管理员工 → 评估排期影响
    │
    ▼ （各子流程独立推进，状态持久化在 LangGraph）
    │
COO 员工：所有子流程完成 → 汇总结果 → 回复客户
```

COO 本身也是无状态的，但 LangGraph 的 Graph 实例记录了"4 个子流程各做到哪了"。

### 6.6 审批流程细粒度设计

不是简单的低/中/高三级，而是**条件审批矩阵 + 会签 + 超时机制**：

```yaml
# 审批规则配置（存储在生产资料层）
approval_rules:
  # ━━━ 按金额分级 ━━━
  financial:
    - condition: "amount < 5000"
      action: auto_approve
      log: true
    - condition: "amount >= 5000 AND amount < 50000"
      approvers: [domain_expert]      # 对应领域专家
      timeout_hours: 48
      timeout_action: escalate_to_ceo
    - condition: "amount >= 50000"
      approvers: [domain_expert, ceo]  # 会签：专家 + CEO 都要批
      timeout_hours: 72
      timeout_action: notify_and_hold

  # ━━━ 按操作类型分级 ━━━
  operations:
    - type: [query, log, notify]
      action: auto_approve             # 查询/记录/通知类直接执行
    - type: [create_draft, generate_report]
      action: auto_approve_with_review  # 自动执行但标记待 review
    - type: [send_to_client, modify_contract]
      approvers: [domain_expert]       # 对外动作需专家确认
    - type: [sign_contract, payment, hire, terminate]
      approvers: [domain_expert, ceo]  # 高影响动作需会签

  # ━━━ 超时与催办 ━━━
  timeout:
    first_reminder_hours: 24
    escalation_hours: 48               # 超时自动升级到 CEO
    max_hold_hours: 72                 # 超过 72 小时冻结流程并告警

  # ━━━ 撤回与变更 ━━━
  change_policy:
    allow_revoke_before_execution: true
    revoke_after_execution: requires_new_approval
    audit_all_changes: true
```

### 6.7 安全与合规设计

#### 数据安全

| 风险点 | 措施 |

- [ ] 部署 OpenClaw，接入企业 IM（飞书/钉钉）
- [ ] **数据集成摸底**：盘点企业现有系统（CRM/ERP/财务/HR/项目管理），评估 API 可用性
- [ ] **数据适配层开发**：为无 API 或 API 质量差的系统开发适配器（这是最耗时的部分）
- [ ] **数据口径统一**：建立实体映射表（客户名/产品名/人员在各系统中的对应关系）
- [ ] 搭建审计日志基础设施
- [ ] 开发 enterprise-data 插件骨架（Tool 注册框架 + 权限隔离）
- [ ] 配置 1 个 Agent（main），验证端到端消息流转
- [ ] **安全评审**：LLM 数据外传风险评估，确定是否需要私有化部署

**交付**：IM 接入可用、至少 2-3 个业务系统 API 打通、安全方案通过评审

**人员**：需要 1-2 名集成工程师，全职投入数据适配

> **现实提醒**：很多中型企业的财务/ERP 系统没有现成 API，可能需要 RPA 或数据库直连方案。仅这一项就可能消耗 1-2 个月。

### 7.2 Phase 1：秘书 + COO（乐观 2 月 / 现实 3-4 月）

**目标**：信息汇总和调度能力上线

- [ ] 配置秘书员工 Profile（日报/周报生成）
- [ ] 配置 COO 员工 Profile（任务分发/查询中枢）
- [ ] 对接企业已有系统 API（项目管理/CRM 只读查询）
- [ ] 实现定时任务（每日自动日报）
- [ ] CEO + 秘书 + COO 三角色跑通
- [ ] **人类 review 机制**：日报生成后先发给指定人员 review，确认准确后再推送全员
- [ ] **质量基线**：连续 2 周日报准确率 > 95% 后，取消 review 环节

**交付**：CEO 可直接查询企业数据，每日自动收到日报

**人员变化**：此阶段不减人，增加 1 名"AI 训练员"负责 review 和纠偏

### 7.3 Phase 2：商务 + 产品研发（乐观 3 月 / 现实 4-6 月）

**目标**：核心业务流程 AI 化

- [ ] 配置商务销售员工 Profile
- [ ] 配置产品研发员工 Profile
- [ ] 实现 CRM 读写、报价流程、合同模板
- [ ] 实现代码辅助、PRD 生成、技术文档
- [ ] 部署 LangGraph 工作流状态引擎，实现多步骤业务流程跟踪
- [ ] 实现审批规则引擎（条件审批 + 会签 + 超时催办）
- [ ] 商务专家 + 技术专家接入验证
- [ ] **并行运行期**：AI 出方案 + 人类同步执行，对比验证 AI 输出质量

**交付**：报价流程端到端 AI 驱动，人类只做关键审批

**人员变化**：行政助理、报表分析岗开始转型为"AI 协作岗"

### 7.4 Phase 3：全职能覆盖（乐观 4 月 / 现实 6-8 月）

**目标**：剩余职能域全部上线

- [ ] 配置项目管理、市场内容、人事行政、财务法务 4 个员工 Profile
- [ ] 对接 HR 系统、财务系统、内容发布平台
- [ ] 实现跨员工协作流程（COO 调度 → LangGraph 并行子流程 → 多员工协同）
- [ ] 完善权限隔离矩阵（按 Agent ID 限制 Tool 访问）
- [ ] 全量日报/周报覆盖所有职能域
- [ ] **合规审计**：财务/法务相关操作接入合规审查流程

**交付**：10 个职能域全覆盖，企业日常运营主要由 AI 驱动

**人员变化**：信息搬运类岗位开始缩减，转岗为现场执行或 AI 运维

### 7.5 Phase 4：优化与自治（第 12-18 月+）

**目标**：提升自治程度，渐进缩减人类介入

- [ ] 基于历史审批数据，低风险操作自动放行（需积累 3-6 个月审批数据）
- [ ] 数字员工间的自主协调（减少 COO 中转）
- [ ] 知识库持续积累，减少重复提问
- [ ] 监控和告警体系（异常行为自动报警、AI 输出质量评分）
- [ ] **渐进减人**：按岗位逐步缩减，而非一次性裁撤

### 7.6 总时间线对比

| Phase | 乐观估算 | 现实估算 | 主要风险 |
|-------|---------|---------|---------|
| Phase 0 | 2 月 | 3-4 月 | 数据集成复杂度 |
| Phase 1 | 2 月 | 3-4 月 | 日报数据质量 |
| Phase 2 | 3 月 | 4-6 月 | 审批流程验证周期 |
| Phase 3 | 4 月 | 6-8 月 | 财务/HR 系统对接 |
| Phase 4 | 持续 | 持续 | 审批自动化需数据积累 |
| **总计** | **~12 月** | **~18-24 月** | — |

### 7.7 人员过渡计划

人员缩减不是一步到位，而是随 Phase 渐进：

```
         Phase 0-1        Phase 2         Phase 3         Phase 4+
         (增效期)        (并行期)        (替代期)         (自治期)
─────────────────────────────────────────────────────────────────
人数      不减反增        开始转型        逐步缩减         趋近终态
          (+1-2 AI运维)   (转岗培训)      (自然减员)       (5-10核心)

关键      验证可行性      AI与人并行       AI主导           AI自治
动作      建立信任        对比输出质量     人类做兜底        人类做决策
```

> **重要原则**：Phase 1-2 期间人员不会减少，反而需要增加"AI 运维/训练"岗位。只有当 AI 输出质量经过充分验证后（Phase 3+），才开始渐进缩减人力。贸然减人会导致质量失控和组织抵触。

## 九、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| AI 幻觉导致错误决策 | 高 | 关键操作必须人类审批；财务数据交叉验证；Phase 1-2 人工 review 全部输出 |
| 业务系统 API 不稳定/缺失 | 高 | Tool 层做重试和降级；无 API 系统用 RPA 或数据库直连；Phase 0 充分摸底 |
| 数据安全与 LLM 外传 | 高 | 敏感字段自动脱敏；评估私有化部署；Tool 级权限隔离矩阵；全量审计日志 |
| Prompt Injection 攻击 | 高 | Tool 输入校验 + 输出过滤；Agent 权限最小化；异常调用自动告警 |
| 数据口径不统一 | 中 | Phase 0 建立实体映射表；数据适配层统一字段名和格式 |
| 员工抵触 | 中 | 先增效再替代；Phase 1-2 不减人；转岗培训；让员工参与 AI 训练 |
| 跨角色任务协调失败 | 中 | LangGraph 并行子流程 + 超时自动升级；COO 兜底协调 |
| 审批流程阻塞 | 中 | 超时催办（24h）→ 自动升级（48h）→ 冻结告警（72h） |
| 单点故障 | 中 | 无状态设计天然支持重启恢复；LangGraph 状态持久化到数据库 |
| 上下文窗口不够 | 低 | 渐进式披露；上下文压缩；只加载当前任务相关数据 |
| 客户不接受 AI 对接 | 中 | 对外沟通仍由人类现场执行者负责；AI 做幕后准备工作 |
| 合规要求人类签字 | 中 | 需特定资质的操作（财务审计、法律意见）保留人类专家全程参与 |

## 十、总结

### 核心公式

```
数字员工 = 通用 Agent 引擎（无状态）
         + Profile 配置包（定义"我是谁"）
         + 生产资料（持久化的业务数据，通过 Tool 读写）
         + Skill（操作流程指导）
         + 每日 Routine（内置 SOP，heartbeat 驱动）
         + 产物体系（交付物 + 累积资产 + 自迭代产物）
         + 时间衰减（热/温/冷分级归档，防止数据垃圾场）
```

### 关键数字

| 指标 | 数值 |
|------|------|
| 数字员工数量 | 8 个（2 管理层 + 6 执行层） |
| 覆盖职能域 | 10 个（全部） |
| 需要的人类核心团队 | 终态 5-10 人（1 决策 + 3-5 专家 + 若干现场）；过渡期不减反增 |
| 实施周期 | 乐观 12 月 / 现实 18-24 月（4 阶段 + 持续优化） |
| 技术栈 | OpenClaw + LangGraph + CrewAI + 自建持久层 |

### 与传统方案的本质区别

```
传统方案：造 8 个独立的 AI 系统，各自维护状态
    ↓
本方案：1 个通用引擎 + 8 套配置 + 统一生产资料层
    ↓
核心优势：
  ✅ 新增角色只需新增一套 Profile，无需开发新系统
  ✅ 生产资料统一管理，天然支持跨角色数据共享
  ✅ Agent 无状态 + 流程有状态，兼顾弹性和业务连续性
  ✅ 渐进式实施，每个 Phase 独立可交付

关键风险认知：
  ⚠️ 数据集成是最大的工程瓶颈，不是 Agent 本身
  ⚠️ 人员缩减必须渐进，过渡期需要增加 AI 运维岗
  ⚠️ 安全合规是企业准入门槛，必须在 Phase 0 解决
  ⚠️ 现实时间线 18-24 月，而非理想的 12 月
```

## 附录 A：本体论语义层（可选增强）

> 以下方案为可选增强，适用于企业系统碎片化严重、需要大量跨系统关联查询、或从零建设业务系统的场景。大多数情况下，主方案（Tool 直连 + events 表）已经够用。

### 何时考虑引入本体层

| 信号 | 说明 |
|------|------|
| 5+ 个异构系统需要交叉查询 | Agent 频繁需要串联 3 次以上 Tool 调用才能回答一个问题 |
| 数据口径混乱 | 同一客户在 CRM 叫"客户A"、财务叫"A公司"、合同叫"A科技有限公司" |
| 从零建设部分业务系统 | 企业缺少项目管理/CRM/工作日志系统，需要自建 |
| 需要复杂图关系分析 | "这个客户关联了哪些项目、哪些人、哪些合同"需要一次性获取 |

### 推荐方案：PostgreSQL + Hasura

不引入额外框架，用已有技术栈实现轻量本体层：

```
PostgreSQL（统一数据库）
├── 业务实体表（表 = Object Type，外键 = Link）
├── events 表（时间层）
└── Hasura 自动生成 GraphQL API + 行级权限 + Actions

Agent Tool 从多个 per-system Tool 简化为 2 个通用 Tool：
  ontology_query  → 一次 GraphQL 查询关联多实体
  ontology_mutate → 统一写操作 + 自动事件记录
```

### 与主方案的关系

本体层不替代 Tool 直连，而是在两种情况下补位：

```
系统有可用 API  → 继续用 Tool 直连（简单直接）
系统无 API 或需自建 → 数据存 PostgreSQL，Hasura 暴露 API
需要跨系统关联 → 两种方式都行：
  · Agent 多次 Tool 调用自行拼接（够用）
  · 或数据同步到 PostgreSQL 后一次 GraphQL 查询（更优雅）
```

### 进阶：Palantir 五层模型的对应关系

参考 Palantir Ontology 的五层数学架构（存在层→关系层→时间层→逻辑层→智能层），在本方案中的对应：

```
Layer 5 智能层  → LLM Agent + QMD 向量记忆（已有）
Layer 4 逻辑层  → Skill 文件 + 审批规则配置（已有）
Layer 3 时间层  → events 事件流水表（4.6 节）
Layer 2 关系层  → Tool 多次调用 / Hasura 外键关联（已有/可选）
Layer 1 存在层  → Tool 对接各系统 / PostgreSQL 统一存储（已有/可选）
```

五层中真正需要额外建设的只有 **Layer 3 时间层**（events 表），其余四层已被主方案覆盖。
