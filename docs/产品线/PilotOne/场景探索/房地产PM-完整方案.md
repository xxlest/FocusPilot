# 房地产项目 PM 操盘助手 · 完整方案 v3

> 本文基于 `房地产PM.md` 原方案复盘重构而成。第一部分逐条展开原方案存在的 10 个问题，第二部分给出重构方案（含角色分层、北极星指标、架构、Agent、数据底座），第三部分是分阶段落地路径。
>
> 文档日期：2026-04-08 · 版本：v3

---

## 第一部分 原方案问题逐条展开

原方案在岗位理解、克制程度上都做得很好——正确地拒绝了"超级驾驶舱""数据中台""自动定价"等重平台幻觉，把 PM 的工作压缩成"盯盘/盯人/盯渠道/盯客户/盯汇报"五件事并逐一对应到 Agent，这是它最大的优点。

但方案存在 **10 条**结构性问题，下面逐条展开。前 8 条是功能和架构层面的，后 2 条是更底层的角色和价值度量缺失——恰恰是决定这个产品能不能被管理层买单的关键。

### 问题 1：数据接入层完全缺失（最致命）

**原方案的表述**："读取现有 Excel、CRM、渠道报备表、销售跟进记录和群消息"

**问题所在**：这一句话把所有脏活都掩盖了。真实项目里：

- **CRM 没有标准**：有的项目用明源云客，有的用思源，有的用自建，有的干脆只有 Excel；字段命名、必填策略、更新频率完全不同
- **渠道报备表是邮件/微信/纸质混合物**：多数项目由案场秘书手工维护，格式每周都在变
- **跟进记录是一句话流水账**：大量是"已联系""客户考虑中""下周再约"，结构化程度极低
- **群消息无法直接读**：企业微信/钉钉需要管理员授权 API，个人微信更是灰色地带
- **录音转写要花真金白银**：一个项目一周通话数千条，哪怕只转销售-客户对话，成本和合规都不是小事

**后果**：如果数据接入层不解决，所有 Agent 产出的都是"垃圾进垃圾出"。这是方案的隐藏命门。

**本质**：原方案的作者把 AI 产品做成了"假设数据现成"的上层应用，但在房地产项目这个场景里，**数据治理本身就是 50% 的工作量**。

---

### 问题 2：Agent 之间没有状态共享

**原方案的表述**：6 个 Agent 各自独立定义输入输出，互不通信。

**问题所在**：PM 的判断本质上是**多维度交叉**的。客户抗性 Agent 发现本周 Top1 抗性是"价格太高"、渠道复盘 Agent 发现渠道 A 来量多但成交差——这两条结论合起来才是真相："渠道 A 客群对价格更敏感，不是渠道质量差，是客群错配"。但原方案里两个 Agent 各跑各的，PM 要自己在脑子里做合流——这恰恰是 AI 最该帮他做的事。

**后果**：每个 Agent 的产出都是"单维结论"，看起来都对但都不够决策。PM 用两周后会说："不如我自己看 Excel。"

**本质**：缺一个"项目记忆"层——一个轻量的、跨 Agent 共享的结构化状态空间。

---

### 问题 3：跟进督办 Agent 会快速变成骚扰机器

**原方案的表述**："每天 11:30、16:30 扫描未首联、未复访、超时未更新客户 → 推送到责任人 → 超过阈值再升级提醒 PM"

**问题所在**：这是此类产品**最经典的死亡路径**。原方案没有规定：同一客户同一催办多久内不重复？销售回复"在联系了"之后怎么静默？PM 连续忽略 N 次提醒后要不要自动降级？周末节假日要不要停？没有这些规则，结果是：

- **第一周**：销售觉得新鲜
- **第二周**：销售开始烦，PM 被抱怨"AI 太烦了"
- **第三周**：大家一起在群里吐槽，然后关掉通知
- **第四周**：项目下线

**本质**：提醒类产品的价值 = 提醒的准确性 × (1 − 噪音比)。原方案只做了前者，没做后者。

---

### 问题 4：竞品快报的数据源完全不现实

**原方案的表述**："定时收集竞品价格变动、促销政策、开盘节奏、主推户型、物料话术变化"

**问题所在**：竞品的核心情报**基本都不公开**——价格变动不会写在官网、促销政策多是"一客一议"、物料话术要靠实地拿资料、开盘节奏会临时调整。AI 能做的**只有整理**，不能做**收集**。原方案把两者混为一谈。

**后果**：Agent 每周产出一份"从网上爬来的公开信息摘要"，PM 一看："这些我早知道了。"

**本质**：边界没划清。竞品情报的核心输入永远是人，AI 的价值是让这些碎片化输入**可以被沉淀、被对比、被检索**。

---

### 问题 5：没有失败兜底和人工覆盖机制

**原方案的表述**：全文没有提及"如果 Agent 判断错了怎么办"。

**问题所在**：房地产营销是**高决策成本**场景。一个错误的"降量渠道 B"建议可能直接导致当月少成交 2 套，损失几十万。PM 对任何自动化工具的信任阈值极高——**只要错一次，就再也不信了**。原方案缺失置信度标注、一键标错、异常时的硬错处理、决策对账。

**后果**：PM 会用一段时间后默默把 AI 结论当参考，然后逐渐忘记它存在。

**本质**：高风险场景的 AI 产品，**人工覆盖和信任建设机制**和 AI 能力本身同等重要。

---

### 问题 6：Skills 和 Agent 边界模糊，存在过度拆分

**原方案的表述**：8 个 Skill（日报读取/渠道表读取/指标计算/异常检测/未成交原因分类/竞品监测/汇报生成/消息提醒）。

**问题所在**：其中"指标计算""异常检测""未成交原因分类"**根本不是独立 Skill**，而是 Agent 内部步骤。拆出来只会带来调用链变长、无状态化、维护成本上升。

**本质**：Skills/Agents/Workflows 的分层没有按"复用性"来划，而是按"功能名词"来划。真正该被拆成 Skill 的是**跨 Agent 复用的、有明确输入输出契约的、无状态的**能力。

---

### 问题 7：没有多项目视角

**原方案的表述**：全文隐含"一个 PM 对应一个项目"的假设。

**问题所在**：真实组织里有单盘 PM、多盘 PM、区域营销总监、集团营销中心四层，原方案在单盘场景下可用，但一旦涉及跨项目对比，整个架构都要重做。

**本质**：数据模型没有考虑多租户/多项目维度，这是架构层面的债。

---

### 问题 8：甲方汇报 Agent 的方向反了

**原方案的表述**："自动把一周数据整理成周报"

**问题所在**：甲方汇报的**核心从来不是数据整理**，而是**政治判断**——这周真实出问题了要讲多直白？上周答应的动作没做到怎么包装？甲方最近换了分管领导口径要不要调？这些全是人的判断。原方案的"AI 生成 → PM 改"路径会让 PM 发现改的幅度太大，最后放弃使用。

**正确方向**：反过来——"PM 列 3 个要讲的点 → Agent 填充数据、话术、图表"。AI 做弹药手，PM 做决策者。

---

### 问题 9：角色分层完全缺失

**原方案的表述**：全文只有"项目 PM"一个用户。

**问题所在**：房地产营销是多角色协作场景——**销售**是数据生产者和执行者、**项目 PM** 是一线操盘、**区域总监**要管 5-10 个项目、**集团营销中心**要看全国 50+ 项目。原方案让所有角色复用 PM 层的产出，会出现两个致命后果：

- **销售视角缺失**：销售只会被动收到催办，没有任何让他觉得"AI 对我有用"的设计——这是在把数据生产者变成产品的敌人
- **管理层视角缺失**：区域总监打开看到 10 个项目各自的"晨会异常"等于没看，他需要的是**跨项目聚合**。原方案没有 Dashboard，没有分数，没有排序，没有红线预警

**本质**：产品只为一个角色设计，注定无法在组织中扩散。

---

### 问题 10：没有北极星指标，尤其是财务维度

**原方案的表述**：全文围绕"异常、提醒、建议"等事件流，没有任何状态指标。

**问题所在**：管理层和老板打开产品第一眼想看到的，**从来不是事件流**，而是几个能直接回答业务问题的数字：

- **这个项目到底赚没赚钱？**（当前盈亏现状）
- **最终会赚多少 / 亏多少？**（按当前去化速度的最终盈亏预测）
- **卖得动吗？会不会烂尾？**（去化速度 vs 预期）
- **增长在哪里 / 风险在哪里？**（趋势和预警）

这些都是**财务问题**，但原方案里完全没有财务数据的位置——所有 Agent 都在算转化率、到访率、跟进时效，没有一个 Agent 算"这个月项目到底赚了多少、剩余货值能不能覆盖成本"。

**后果**：项目老板、财务总监、区域总打开产品后第一反应是："这是给销售经理用的吧，和我没关系。"——然后产品就失去了向上推广的空间。

**本质**：原方案是"执行视角"的工具，不是"经营视角"的工具。而真正决定产品能不能活下来的，恰恰是能不能回答经营视角的问题。**一个不能告诉老板"赚没赚钱"的房地产 AI 产品，在组织里活不过 3 个月**。

## 第二部分 重构方案

### 2.1 产品定位（收紧版）

**一句话**：给房地产项目团队的**分角色经营助手**——对销售是梳理，对 PM 是弹药手，对管理层是北极星仪表盘。

**三个克制边界**：

- **不替人做决策**，只把决策需要的信息、数据、话术、趋势准备好
- **不做数据中台**，只做"读既有、不要求新录入"的轻量层
- **不追求覆盖所有岗位场景**，优先解决各角色每天/每周最消耗时间的 Top 1-3 件事

**和原方案的最大区别**：

1. 从"一个 PM 工具"升级为"**角色分层的经营助手**"
2. 新增"**财务北极星**"作为管理层入口，让产品能向上推广
3. 从"6 Agent 一次性铺开"改为"**3 阶段渐进，每阶段解锁一个新角色**"

---

### 2.2 角色分层与信息密度

这是方案的**总纲**。其他所有设计都要回答"这东西服务哪个角色、在哪一层"。

#### 2.2.1 四类角色 × 核心诉求

| 角色 | 每日/每周核心痛点 | 需要的信息密度 | 核心触达形式 |
|------|------------------|----------------|--------------|
| **销售** | 今天该联系谁、有什么能帮我、怎么应对客户抗性 | 个人客户颗粒度 | 钉钉/企微个人消息 + 客户画像卡 |
| **项目 PM** | 今天项目哪里异常、这周怎么向甲方汇报、销售团队状态 | 项目颗粒度 | 钉钉推送 + 日报/周报文档 |
| **区域总监** | 我管的 5-10 个项目谁红了、谁该救火、资源怎么调 | 项目聚合（跨项目） | **Dashboard** + 周报 |
| **集团营销中心 / 老板** | 全公司赚没赚钱、最终预测盈亏、系统性风险、定价要不要调 | 全局聚合 + 财务 | **Dashboard**（北极星仪表盘）+ 月度简报 |

#### 2.2.2 关键设计原则

1. **不同角色看不同东西，不是同一份产出换个皮肤**
   - 销售看的是"我的今日重点"（个人级）
   - PM 看的是"本项目异常和今日重点"（项目级）
   - 区域总看的是"10 个项目健康度排名 + Top3 风险项目"（聚合级）
   - 老板看的是"4 个北极星分数 + 盈亏预测"（财务级）

2. **信息往上走要做聚合和过滤，不能原样透传**
   - 销售级的客户抗性 → PM 级要聚合成 Top3 抗性 → 区域级要聚合成跨项目共性 → 集团级要聚合成趋势和定价启示
   - **每上一层，信息密度降 10 倍，决策价值升 10 倍**

3. **越往上的角色，信任阈值越高，上线越晚**
   - 销售容错高（反正只是推送建议），Phase 2 上
   - PM 容错中（是他的工作被替代），Phase 1 上（但要建信任曲线）
   - 区域总监容错低（影响决策），Phase 3 上
   - 集团/老板容错极低（一次不准就永久关闭），**Phase 3 下半段才上**

4. **权限隔离**：销售只看到自己的数据，PM 只看到自己的项目，区域总只看到自己管的项目——这是企业级产品的基本盘。

---

### 2.3 北极星指标体系（管理层 Dashboard 的灵魂）

这是原方案完全缺失的维度。北极星指标的设计标准是：**管理层打开产品第一屏看到这几个数字，就能回答"我这个项目/区域/公司到底怎么样"**。

#### 2.3.1 四个北极星（只要四个，不多不少）

| 北极星 | 回答什么问题 | 核心组成 |
|--------|-------------|---------|
| **① 赚钱了吗**（盈利现状） | 这个项目到现在为止赚了多少 / 亏了多少 | 累计销售额、累计回款、已结算成本、当前毛利率、现金流净值 |
| **② 会赚多少**（盈利预测） | 按当前速度跑到最后是赚是亏 | 剩余货值、去化速度、预测售罄时间、预测最终售价、预测最终毛利、**预警亏损阈值** |
| **③ 卖得动吗**（去化与执行） | 销售节奏健康吗、团队在不在状态 | 月度去化率、转化漏斗健康度、A 类客户转化率、执行力指数 |
| **④ 定位准吗**（客群匹配） | 产品定价和市场有没有错位 | 抗性结构变化、渠道客群匹配度、竞品价差、退订率趋势 |

**关键说明**：

- **① 和 ② 是财务北极星**，直接回答老板的问题："赚没赚钱、会赚多少、会不会亏"
- **③ 和 ④ 是执行北极星**，回答区域总的问题："卖得动吗、定位准吗"
- **每个北极星产出一个 0-100 的分数 + 一个趋势箭头 + 一个 Top3 异动提示**

#### 2.3.2 财务北极星的数据来源（重点）

财务数据通常不在销售 CRM 里，而在**财务系统 + 成本系统 + 回款系统**，这是原方案完全没涉及的。方案处理策略：

1. **只读财务系统月度快照**（不碰实时数据）
   - 累计销售额 / 累计回款（从财务系统月末对账表）
   - 已发生成本 / 预算成本（从成本系统月度快照）
   - 剩余货值（从项目主数据）
2. **预测模型走极简路线**
   - 不做复杂计量经济模型，只做两条曲线外推：
     - 去化速度外推 → 预测售罄时间
     - 当前均价 × 剩余货值 − 剩余成本 → 预测最终毛利
   - 预测值附**区间**（乐观/中性/悲观），不给单点数字
3. **预警线硬卡**
   - 预测毛利率 < 5% → 橙色预警
   - 预测毛利率 < 0 → 红色预警（告诉老板"这个项目会亏"）
   - 预测售罄时间 > 24 个月 → 流动性预警（告诉老板"这个项目会压资金"）

**这三条预警就是产品最核心的向上价值**——能让老板在月度会上花 30 秒知道哪些项目该救火。

#### 2.3.3 Dashboard 第一屏设计原则

打开 Dashboard 第一屏**只显示**：

1. 4 个北极星分数（大数字 + 趋势箭头 + 颜色）
2. Top5 红线项目排序（按预警严重度）
3. 本月关键数字 3 个（总销售额、总回款、预测最终盈亏）

**不显示**的：原始数据表格、单项目明细、具体抗性分类——这些都在下钻里，默认不出现。

**管理层看 Dashboard 的时间预算是 2 分钟**，超过 2 分钟就失败了。

6. **新增经营视角 Agent 组**：解决问题 10（财务北极星）

### 2.5 Agent 设计（按角色分组）

#### PM 组（Phase 1）

##### Agent ①：晨会准备 Agent

**价值**：PM 每天开晨会前拼表格的痛点。

**输入**：昨日销售跟进表、渠道报备表、成交/认购数据、项目记忆层。

**输出**（严格 4 段）：

1. **昨日结果**：认购 X 套、签约 Y 套、到访 Z 组（附环比）
2. **3 条异常**（没有异常时说"昨日平稳"，不硬凑）
3. **今日重点**：重点盯谁、重点跟哪个渠道、重点催谁
4. **甲方可能问的问题**（从项目记忆层取最近甲方关注点）

**产出时间**：每天 7:30 推送钉钉/企微
**信任机制**：异常标注置信度 + 一键标错 → 项目记忆层降权

---

##### Agent ②：跟进督办 Agent

**价值**：PM 下午反复催销售的痛点。

**触发规则**（含静默机制）：

| 规则 | 说明 |
|------|------|
| **去重窗口** | 同一客户同一催办 24 小时内不重复 |
| **静默反馈** | 销售一键回复"在联系了/明天再约/客户无效"，对应 6h/24h/永久静默 |
| **降级机制** | PM 连续 3 次未点开某销售的提醒 → 自动降为每日合并一条 |
| **节假日** | 周末/法定假日停发，除非 PM 手动开启 |
| **升级阈值** | 仅"A 类客户 > 48h 未跟进"或"承诺今日回电未执行"升级给 PM |

**产出时间**：每天 11:30 和 16:30（可配置）
**关键设计**：宁可少提醒，不可多提醒。PM 的信任是**一次性的**。

---

##### Agent ③：周报/甲方汇报 Agent（方向反转）

**核心变化**：不自动选题，做弹药手。

**交互流程**：

1. 周五 10:00，Agent 推送"本周数据概览"（纯事实，不做判断）
2. PM 在对话框里列出 3-5 个要讲的点（如"重点说渠道 B 为什么突然好"、"弱化签约率下滑"）
3. Agent 根据 PM 选题**填充数据、话术、图表**
4. 输出两份草稿：内部版（详）+ 甲方版（简）
5. PM 二次修改后定稿

**内部版 vs 甲方版的固定规则**：

- 内部版：讲执行问题、归因、下周动作、需要的资源
- 甲方版：讲结果、里程碑、下周计划，弱化内部矛盾

**不做的事**：不自动选题、不替 PM 做政治判断、不生成 PPT。

---

#### 销售组（Phase 2）

##### Agent ④：我的今日重点 Agent

**价值**：让销售觉得 AI **对我也有用**，从被催的对象变成被服务的对象。

**输入**：我的客户池、我的跟进记录、昨日约访、项目记忆层中的话术库。

**输出**（每天 8:30 推送个人钉钉）：

1. **今日优先联系 3 人**：从我的 A 类客户里挑出需要今天联系的
2. **今日约到访提醒**：昨天约了今天来看的客户有几组
3. **待跟进客户**：昨天的咨询今天该回的
4. **1 条话术建议**：针对本周高频抗性的应对话术

**关键设计**：

- **只梳理，不催办**——这是"为我服务"和"被我讨厌"的唯一区别
- 销售可以一键"这个不准"，反馈直接进项目记忆层（既帮销售又帮 AI 学习）

---

##### Agent ⑤：客户画像 Agent

**价值**：销售点任何一个客户名字，3 秒看到完整画像。

**输出**（弹出式卡片）：

1. **基础信息**：联系方式、首访日期、意向户型、预算
2. **跟进摘要**：最近 3 次跟进的 AI 归纳（不是原文堆砌）
3. **抗性标签**：AI 从跟进记录里提取的抗性（价格/户型/地段...）
4. **推荐话术**：针对该客户抗性的话术建议
5. **下一步建议**：AI 建议的下一步动作

**关键设计**：**读只读，不写回 CRM**——避免侵入现有流程。

---

#### 深度分析组（Phase 2）

##### Agent ⑥：客户抗性 Agent

每周输出本周"客户为什么不买"的 Top3 抗性结构化结论，不是客户原话堆砌。先用**项目自己的标签体系**，不做通用分类。

##### Agent ⑦：渠道复盘 Agent

每晚输出"保留/加码/降量/暂停"建议，而非大报表。结合项目记忆层里的客户抗性做合流判断（如"渠道 A 客群对价格更敏感"）。

##### Agent ⑧：竞品整理 Agent（整理而非收集）

**核心变化**：不做自动爬虫，做**沉淀工具**。PM 或销售随手转发一条竞品信息（价格变动/促销/新物料），Agent 自动归类入库。每周一产出"本周竞品动态汇总"。

---

#### 经营视角组（Phase 3，北极星驱动）

##### Agent ⑨：项目健康度 Agent

**价值**：给区域总监一个"我管的项目谁红了"的快速筛选。

**输入**：北极星四维数据（盈利现状/盈利预测/去化执行/客群匹配）。

**输出**：

1. **项目健康度分数**（0-100，加权计算）
2. **Top 风险项**：本项目最大的 3 个风险点（按严重度）
3. **对比基准**：同区域/同类型项目的分数排名

**产出时间**：每日凌晨批量计算，区域总监每天 8:00 看 Dashboard 即可。

---

##### Agent ⑩：财务预测 Agent（最关键）

**价值**：直接回答老板的 3 个问题——"赚了吗、会赚多少、会不会亏"。

**输入**：

- 财务系统月度快照（累计销售额、累计回款、已发生成本）
- 成本系统（预算成本、剩余成本）
- 销售去化数据（本月去化率、剩余货值、均价趋势）

**输出**（月度更新，紧急情况周更）：

1. **当前盈亏状态**：毛利率、现金流净值、是否已盈亏平衡
2. **最终盈亏预测**：给出乐观/中性/悲观**三档区间**
   - 乐观：按近 1 个月最快去化速度和均价
   - 中性：按近 3 个月平均
   - 悲观：按近 3 个月最慢速度 + 均价下浮 5%
3. **预警等级**：
   - 🟢 正常：预测毛利率 > 10%
   - 🟡 关注：5% < 预测毛利率 < 10%
   - 🟠 橙色：0 < 预测毛利率 < 5%
   - 🔴 红色：预测毛利率 < 0（项目会亏）
   - ⚫ 黑色：预测售罄时间 > 24 个月（流动性风险）
4. **核心归因**：如果恶化，是因为去化慢了、均价降了、还是成本超了

**关键设计**：

- **不给单点数字**，永远给区间 —— 财务预测的不确定性不能被 AI 藏起来
- **预警规则硬编码**，不由 LLM 自主判断 —— 这类决策不能让模型临场发挥
- **月度节奏为主**，不追求日更 —— 财务数据本身就是月度口径
- **附一张简表**："如果想让预测从红色变橙色，需要 X 个月内多卖 Y 套、均价不低于 Z"

这个 Agent 是整个产品能否向上推广的**命门**。做好了，老板愿意每月花 5 分钟看它；做不好，产品永远只是销售经理的工具。

---

### 2.6 项目记忆层

**目标**：让 10 个 Agent 共享状态，形成合流判断。

**数据结构**（极简）：

```yaml
project_memory:
  project_id: "xxx花园"

  current_week:
    top_resistances: # 客户抗性
      - { label: "价格太高", count: 23, trend: "上升" }
      - { label: "户型不喜欢", count: 12, trend: "平稳" }
    channel_status: # 渠道状态
      - { name: "渠道A", quality: "待观察", reason: "来量多但转化差" }
    sales_focus: # 销售重点
      - { name: "张三", issue: "A 类客户跟进超时 2 例" }

  current_month:
    financial_snapshot: # 财务快照（Phase 3）
      cumulative_sales: 2.3e8
      cumulative_collection: 1.9e8
      gross_margin: 0.08
      predicted_final_margin: [0.03, 0.07, 0.11] # 悲观/中性/乐观
      alert_level: "orange"

  recent_4_weeks: # 时间序列
    conversion_rates: [...]
    top_resistances_trend: [...]

  recent_12_months: # 长周期（Phase 3）
    monthly_financial: [...]

  pm_annotations: # PM 手动标注（最高权重）
    - date: "2026-04-01"
      type: "agent_wrong"
      content: "渠道 B 降量建议不对，是上周活动影响"

  owner_concerns: # 甲方最近关心
    - { date: "2026-04-05", topic: "签约率" }
```

**关键原则**：

- 只存**结构化结论**，不存原始数据
- PM 标注权重最高，Agent 产出次之
- 每周/每月自动归档一次，防止膨胀

### 2.7 数据接入层（方案基石）

**核心动作：数据源盘点表**

每上一个新项目必须先做 1 周数据盘点，产出一份表（按数据类型分销售类和财务类）：

| 类型 | 数据源 | 格式 | 更新频率 | 字段质量 | 接入方式 | 风险 |
|------|--------|------|----------|----------|----------|------|
| 销售 | 销售跟进 | Excel 每日导出 | 每天 18:00 | 中 | 文件监听 | 文件命名不规范 |
| 销售 | 渠道报备 | 企微群消息 | 实时 | 低 | 人工整理后导入 | 需增加秘书工作量 |
| 销售 | 成交签约 | 明源 CRM | 实时 | 高 | API 只读 | 需 IT 授权 |
| 销售 | 客户录音 | —— | —— | —— | **不接入** | 成本/合规风险 |
| 财务 | 累计销售额 | 财务系统 | 月末 | 高 | 月度快照 | 需财务部门授权 |
| 财务 | 已发生成本 | 成本系统 | 月末 | 高 | 月度快照 | 需成本部门授权 |
| 财务 | 回款进度 | 回款系统 | 周 | 中 | 周度快照 | 回款口径可能混乱 |
| 财务 | 剩余货值 | 项目主数据 | 月 | 高 | 月度拉取 | 调整时需同步 |

**原则**：

- **只做"读既有"**，不要求销售或秘书新录入
- **财务数据只做月度快照**，不追求实时
- **接入不了的数据源直接标红**，Agent 产出时注明"此部分缺失"
- **数据质量监控**：每日跑一次字段缺失率检查，异常时推送告警而非硬凑结论
- **权限分离**：财务数据只给 PM 以上角色看，销售看不到

---

### 2.8 反馈闭环与信任建设

**三个触点**：

1. **即时反馈**：每条 Agent 产出附一键"不对"按钮，标注原因（"数据错"/"结论错"/"不重要"）
2. **周度对账**：每周五在周报里附"本周 AI 产出 vs 实际决策的一致率"
3. **月度复盘**：每月一次 PM/区域总和产品团队对话，调整规则

**信任曲线分角色建设**：

| 角色 | 第 1 周 | 第 2-3 周 | 第 4 周起 |
|------|---------|-----------|-----------|
| PM | 只推送，不要求反馈 | 开反馈按钮 | 公布一致率，目标 > 80% |
| 销售 | 只推送 | 开反馈按钮 | 一致率 > 85%（销售容忍更低） |
| 区域总监 | 每周只看 1 次 Dashboard | 开始用 | 一致率 > 90% |
| 集团/老板 | **不上线** | **不上线** | 前三个月必须 > 95%，错一次就停 |

**核心原则**：越高层的角色，上线门槛越高，错一次的代价越大。

## 第三部分 分阶段实施

### 总体节奏

```
Phase 0 ──> Phase 1 ──> Phase 2 ──> Phase 3
数据盘点     PM 最小闭环    销售+深度分析   经营视角+管理层
(不写代码)   (1 种子项目)   (1-2 项目)     (多项目+北极星)
```

**核心理念**：**每个 Phase 解锁一个新角色**。不是一次性做完，而是用 Phase 1 建 PM 信任、Phase 2 扩展到销售和深度分析、Phase 3 才向上打管理层。

---

### Phase 0：数据盘点 & 试点选型（不写代码）

**目标**：选定 1 个种子项目，摸清数据底细。

**核心动作**：

1. **选种子项目**：
   - PM 愿意配合（最重要）
   - 在售阶段（不是尾盘或蓄客期）
   - CRM 数据相对规整
   - 规模适中（月成交 15-30 套）
   - **财务数据可获取**（为 Phase 3 做准备）

2. **蹲点 1 周**：
   - 跟 PM 一天完整流程
   - 拿到所有数据源样本（含财务类）
   - 填写数据源盘点表
   - 识别 PM 最耗时的 Top 3 动作

3. **产出物**：
   - 数据源盘点表（含风险清单）
   - PM 日常动作时间分布表
   - Phase 1 的"不做清单"

**退出条件**：数据盘点表完成，PM 口头确认方向。

---

### Phase 1：PM 最小闭环

**目标角色**：只开放给项目 PM。**销售、区域总、老板全部不上线**。

**交付内容**：

- **数据接入层**：种子项目的 2-3 个销售类数据源适配器
- **项目记忆层**：最小 KV 版本 + 每日快照
- **3 个 PM 组 Agent**：晨会、督办（全静默规则）、周报（弹药手）
- **反馈闭环**：一键标错 + 周度对账

**信任建设节奏**：

- 第 1 周：只推送，观察反应
- 第 2-3 周：开反馈按钮，每日调参
- 第 4 周：公布一致率

**验收标准**（四项硬指标）：

1. 晨会 Agent：PM 每天**在晨会上实际用到**的比例 > 70%
2. 督办 Agent：销售和 PM 都**没有主动关闭通知**
3. 周报 Agent：PM 定稿时**相比 AI 草稿的修改幅度 < 30%**
4. 整体：PM 月末主动说"有用，想继续"

**退出条件**：四项全部达标，且连续 2 周稳定。

---

### Phase 2：扩展到销售 + 深度分析

**前置条件**：Phase 1 验收通过，种子项目 PM 主动要求扩展。

**目标角色**：新增**销售**。区域总和老板仍不上线。

**交付内容**：

1. **销售组 Agent**（2 个）：
   - 我的今日重点 Agent（"为我服务"模式）
   - 客户画像 Agent（只读客户卡）

2. **深度分析组 Agent**（3 个）：
   - 客户抗性 Agent（周频）
   - 渠道复盘 Agent（日频晚间）
   - 竞品整理 Agent（整理而非收集）

3. **项目记忆层升级**：
   - 加时间序列（最近 4 周趋势）
   - 加跨 Agent 合流判断

4. **数据接入层扩展**：
   - 适配更多数据源格式
   - 引入"秘书辅助整理"模式（助理每天花 10 分钟清洗）

5. **信任机制完善**：
   - 建立"AI 产出 → PM 决策 → 实际结果"对账链路
   - 每月一次"AI 判断准确率"报告

**验收标准**：

- 销售主动使用率 > 60%（打开过"我的今日重点"的销售占比）
- 销售反馈数据回流进项目记忆层（每周 > 20 条）
- 8 个 Agent 中至少 6 个被日常使用
- 一致率 > 85%
- 种子项目 PM 愿意为第二个项目推荐同事

---

### Phase 3：经营视角 + 管理层上线

**前置条件**：
- Phase 2 验收通过
- 至少 **2 个独立项目**跑通 Phase 2（避免单项目偏差）
- 财务系统接入已谈妥

**目标角色**：新增**区域总监**（Phase 3a）和**集团/老板**（Phase 3b，更晚）。

#### Phase 3a：多项目复制 + 区域总监 Dashboard

**交付内容**：

1. **数据模型多租户化**：
   - 重构项目记忆层支持多项目隔离
   - 聚合层实现：项目 → 区域聚合
   - Agent 改造为项目无关

2. **项目健康度 Agent（⑨）**：
   - 0-100 分数 + Top3 风险项 + 对比基准

3. **区域总监 Dashboard 第一版**：
   - 我管的项目健康度排名
   - Top5 风险项目
   - 本区域关键趋势

4. **部署工程化**：
   - 新项目接入 SOP（目标：1 周盘点 + 3 天适配）
   - 沉淀 3-5 个常见 CRM 适配器

**验收标准**：

- 新项目接入成本 < 2 人周
- 区域总监**每周打开 Dashboard ≥ 3 次**
- 一致率 > 90%
- 至少 5 个项目在生产运行

#### Phase 3b：北极星 + 财务预测 + 老板上线

**前置条件**：
- Phase 3a 稳定运行 2 个月以上
- 财务数据接入完成并验证准确性
- 至少 8 个项目在生产运行

**交付内容**：

1. **财务数据接入**：
   - 财务系统 / 成本系统 / 回款系统月度快照
   - 权限严格隔离

2. **财务预测 Agent（⑩）**：
   - 当前盈亏状态
   - 最终盈亏预测（三档区间）
   - 五级预警（🟢🟡🟠🔴⚫）
   - 核心归因

3. **北极星 Dashboard**：
   - 四个北极星分数（大字）
   - Top5 红线项目
   - 3 个本月关键数字
   - 2 分钟阅读预算

4. **月度简报**：
   - 自动生成发给集团营销中心
   - 包含预警项目清单 + 建议动作

**验收标准**（最严格）：

- **财务预测准确率**：中性区间与实际月末结果偏差 < 10%
- **预警误报率** < 5%（一次重大误报可能摧毁整个产品）
- 老板/集团营销中心每月至少打开 1 次
- 至少 1 个预警项目被老板主动提及
- 连续 3 个月无重大事故

**这一阶段是整个产品的终极考验**。达标意味着产品从"销售经理工具"升级为"经营决策工具"，商业价值翻一个量级。

| 节点 | 决策 | 判据 |

## 一句话收束

**原方案对一线岗位的理解是对的，但它只是"销售经理的工具"——没有角色分层，没有财务视角，没有向上推广的路径。在组织里这种工具活不过半年**。

新方案的核心升级是三个"反直觉"决定：

1. **分角色渐进上线**——先 PM、再销售、最后管理层。信任曲线不能跳级
2. **把财务预测放在产品的最高点**——四个北极星里 ① 赚钱了吗、② 会赚多少是老板视角的命门
3. **Phase 3b 宁可晚 3 个月也不能仓促**——财务预测一次误报就摧毁整个产品的组织信任

只有这样，项目 PM 才会说"我明天还想用"，销售才会说"这个 AI 对我也有用"，区域总监才会每周打开 Dashboard，老板才会在月会上主动提起其中一个预警项目——**这四句话全部成立的那一天，这个产品才算真正在组织里活下来了**。
