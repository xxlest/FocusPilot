# Pilot 产品家族

> **状态**：规划中。当前仅实现 FocusPilot 本地能力，PilotOne 为企业产品线规划。
> **目的**：说明 Pilot 产品家族的定位、版本边界和长期演进方向。

---

## 1. 产品家族定位

Pilot 产品家族围绕同一个核心判断展开：AI 时代的关键产品形态，不是帮人管理更多事项，而是让人和组织拥有可持续自迭代的执行系统。

**FocusPilot** 面向个人，是个人自迭代 AIOS。它用 Project 组织认知，用 Markdown 保存长期记忆，用 Agent 执行任务，用反馈持续重构个人工作系统。

**PilotOne** 面向企业，是中小企业 AI 化操作系统。它用行业本体和企业实例本体统一建模业务对象、流程、知识和规则，用 PilotOne-Agent 数字员工体系持续执行经营任务，用 OrgBrain 沉淀企业记忆，帮助中小企业低成本完成 AI 化升级。

两条产品线不是包含关系，而是共享底层 Agent Engine，并在不同场景中形成不同产品形态：

- FocusPilot：个人场景，核心是个人项目、知识和执行闭环。
- PilotOne：企业场景，核心是企业本体论、数字员工和组织自迭代。

## 3. PilotOne 的系统命题

PilotOne 的底层假设是：中小企业 AI 化的关键，不是买更多 SaaS，也不是训练一个更聪明的聊天助手，而是把企业变成一个 AI 能理解、能执行、能复盘、能自我更新的系统。

因此 PilotOne 由三层核心能力组成：

| 层级 | 作用 | 说明 |
|------|------|------|
| 行业本体 + 企业实例 | 让 AI 理解企业 | 用可复用的行业概念层承载订单、产品、BOM、供应商、渠道、规则等语义关系，再落到每家企业的私有数据实例 |
| PilotOne-Agent 数字员工 | 让 AI 执行业务 | Hermes-Agent 自迭代引擎在企业场景的产品化实例，每个数字员工拥有职责、状态、权限、Routine、事件输出和协作协议 |
| 自迭代闭环 | 让企业越用越聪明 | Routine 产生 Events，Events 触发规则和 SOP 优化，再按自治级别自动生效、通知或等待审批 |

这使 PilotOne 的价值不止是“自动化几个流程”，而是为中小企业提供一套可持续进化的 AI 组织底座：先从单个高频场景切入，再逐步接入更多业务系统，最终形成企业自己的数字员工团队和业务记忆系统。

PilotOne 的长期目标，是以低成本、可复制、可渐进落地的方式，赋能万家中小企业拥有自己的 AI 组织能力。

## 4. PilotOne 高层架构

```text
                 ┌──────────────────────────────┐
                 │  战略驾驶舱 / IM / Web 入口   │
                 │  目标设定、审批授权、结果验收  │
                 └──────────────┬───────────────┘
                                │
经营目标 ──▶ 行业本体模板 ──▶ 企业实例化本体 ──▶ PilotOne-Agent 执行
                                │                    │
                                ▼                    ▼
                         OrgBrain / Events ◀── Routine / Tool / Skill
                                │
                                ▼
                         Hermes 自迭代引擎
                                │
                                ▼
                      规则 / SOP / Routine 更新
                                │
                                ▼
                 ┌──────────────────────────────┐
                 │ Bridge：ERP / CRM / 财务 / IM │
                 └──────────────────────────────┘
```

这个架构的主轴不是“一个管理后台”，而是“经营目标进入本体，数字员工执行，事件沉淀为组织记忆，再通过 Hermes 自迭代引擎更新规则和 Routine”。战略驾驶舱只是老板与核心员工的入口，Bridge 是连接现实系统的基础设施。

详细设计沉淀在 [PilotOne产品方案.md](产品线/PilotOne/PilotOne产品方案.md)。

## 5. FocusPilot vs PilotOne 的本质区别

| | **FocusPilot** | **PilotOne** |
|---|---|---|
| 产品本质 | 个人自迭代 AIOS | 中小企业 AI 化操作系统 |
| 主要问题 | 个人如何把想法变成项目、任务、执行和知识复利 | 企业如何让 AI 理解业务、持续执行、沉淀经验、更新规则 |
| 用户心智 | “我的 AI 执行系统” | “公司的 AI 组织底座” |
| 执行主体 | 个人授权 Agent 执行 | 数字员工按职责、权限和 Routine 持续执行 |
| 知识系统 | 个人 Markdown workspace | 行业本体 + 企业实例本体 + OrgBrain |
| 自迭代对象 | 个人认知、项目系统、知识结构 | 业务流程、SOP、决策规则、组织能力 |
| 核心循环 | 想法 → Task → Agent 执行 → 个人知识成长 | 目标 → 本体建模 → 数字员工执行 → 事件沉淀 → 规则自迭代 → 组织进化 |

## 7. 商业模式草案

| 版本 | 定价模式 | 参考价格 | 包含 |
|------|----------|----------|------|
| FocusPilot 本地版 | 免费 / 低价买断 | ¥0 或 ¥98 买断 | 本地 macOS 能力，用户自带 API Key |
| FocusPilot 云端版 | 月度订阅 | ¥49/月 或 ¥399/年 | 本地全部能力 + 云端 Engine + 远程访问 |
| PilotOne 场景版 | 按场景包订阅 | 待验证 | 单一高频场景，如客户跟进、内容获客、经营日报 |
| PilotOne 企业版 | 场景包 + 数字员工 + 集成计费 | 场景包优先，数字员工为成本锚点 | 数字员工体系、企业本体论、OrgBrain、系统集成、IM 入口 |

PilotOne 不建议一开始按传统 SaaS 人头计费，也不适合把五个数字员工拆成孤立 SKU。更适合的商业逻辑是以场景包作为最小购买单元，再按数字员工深度、业务场景复杂度和系统集成复杂度扩展，因为客户购买的不是账号，而是一段可运行、可沉淀、可自迭代的企业 AI 化能力。
