# FocusPilot 设计规范

> **版本**：V4.3
> **日期**：2026-05-15
> **读者**：UI 修改、视觉调整、新增界面元素时必读

---

## 1. 设计基调与哲学

### 核心基调：干净 · 高级 · 克制 · 专业

FocusPilot 追求 **macOS 原生级精致 UI**，像系统自带应用一样自然融入桌面。
设计上宁可少做，不可过度——每一个像素、每一帧动画都应有明确理由，拒绝装饰性元素。

### 产品哲学：专注 × 禅意

品牌灵魂来自禅圆（Enso）——一笔未闭合的弧线，画在悬浮的主题渐变球体上。
传递「专注当下」「开放循环」「柔和过渡」三层含义。
所有 UI 决策围绕「专注、轻量、不打断」展开。

## 2. 品牌图标设计：禅圆（Enso）

### 2.1 图标构成

图标由两个核心元素叠加而成：

1. **底层：主题渐变球体** — 由当前主题的 accent 派生径向渐变，带有左上高光、底部暗影和细微内边缘光，营造出立体的「悬浮球」质感
2. **上层：禅圆 Enso** — 一条白色未闭合弧线，从左上 10–11 点方向的厚重起笔开始，顺时针经过顶部和右侧，在右下以三个渐隐散点过渡，并在底部左侧留下几道极细余笔

### 2.2 与 FocusPilot 的寓意关联

| 元素 | 寓意 |
|---|---|
| **禅圆** | 专注当下——一次只进入一个明确的工作状态 |
| **未闭合弧（约 300°）** | 开放循环——任务可以开始、暂停、恢复和调整，而不是僵硬闭环 |
| **左上起笔 → 顺时针运行 → 右下渐隐** | 启动到完成——视觉读法从左上果断投入，经顶部和右侧形成循环，在右下自然减速 |
| **笔触从粗到细** | 能量流动——从全神贯注到柔和收尾，而非戛然而止 |
| **起笔墨滴** | 启动动作——每次点击「开始专注」时的明确决心 |
| **收笔散点（三个渐隐）** | 结束缓冲——对应完成提示、pending 状态和后续动作的柔和过渡 |
| **底部细余笔** | 认知余韵——专注结束后仍保留轻微延续感，不做生硬切断 |
| **悬浮球体** | 常驻陪伴——始终悬浮在屏幕边缘，随时待命但不打断 |

- **修改时必须同步原型资产、App 图标与悬浮球实现**，确保所有入口视觉一致

### 2.4 绘制参数

| 参数 | 值 | 说明 |
|---|---|---|
| 弧线角度 | 285°–300° | 主弧线从左上顺时针覆盖顶部、右侧和下方，开口留在左侧偏下 |
| 起笔位置 | 左上 10–11 点方向 | 起笔墨滴与球体高光形成视觉锚点 |
| 点状收束 | 右下 4–5 点方向 | 三个渐隐散点表达柔和结束 |
| 余笔位置 | 底部左侧 7–8 点方向 | 极细短弧，表现墨迹余韵 |
| 最大笔宽 | `size * 0.095`–`0.105` | 起笔更有力量 |
| 最小笔宽 | `size * 0.012`–`0.020` | 底部余笔足够轻 |
| 透明度范围 | 0.90 → 0.72 | 从起笔到收笔渐降 |
| 缓出曲线 | `1.0 - pow(1.0 - t, 2.0)` | 模拟毛笔力度变化 |
| 墨滴半径 | `size * 0.08`–`0.09` | 起始墨滴略大于当前版本，强调启动感 |
| 散点半径 | 0.032 → 0.020 → 0.012 | 三个渐隐收笔散点，避免过重 |
| 散点透明度 | 0.56 → 0.34 → 0.18 | 渐隐效果 |

## 3. 视觉准则

### 3.1 克制用色

Notion 风格 8 色主题（浅 4 + 深 4），每主题 9 色槽覆盖全 UI。accent 仅用于关键操作和状态指示，大面积留白/留灰，不滥用彩色。

品牌图标在 UI 内使用主题化版本：颜色必须跟随 FocusPilot 原 AppTheme 映射，默认白为克制中国红、暖象牙试用珊瑚红、薄荷绿为绿、淡天蓝为蓝、经典深为亮蓝、深海蓝为海洋蓝、墨绿为亮绿、纯黑为薰衣草紫。主题化只服务界面协调，不改变商标定稿资产。

### 3.2 原生质感

毛玻璃（NSVisualEffectView）、系统动画曲线（ease-out/ease-in）、SF Symbols，不引入自定义图标库。

### 3.3 层次分明

textPrimary / textSecondary / textTertiary 三级文字层次严格区分，信息权重一目了然。

### 3.4 精致细节

悬浮球径向渐变 + 高光 + 暗区 + 内边缘光 + accent 呼吸光晕；hover 缩放 1.06x；贴边吸附动画——细节服务于高级感，而非炫技。

### 4.5 悬浮球颜色派生

悬浮球渐变色由 `AppTheme.ballGradientColors` 从 accent 自动派生：
- `light` = accent.blended(withFraction: 0.3, of: .white)
- `medium` = accent
- `dark` = accent.blended(withFraction: 0.4, of: .black)

### 4.6 V1 FP-UI 主题系统（与 V4 的边界）

V1 FP-UI 原型直接沿用 V4 的 **8 色 AppTheme 主题系统**（浅色 4 + 深色 4），色值与 `Models.swift` 中的 `ThemeColors` 一一对应。原型通过 `data-theme` 属性切换 CSS 变量实现即时预览，默认选中经典深（classicDark）。`08-settings.md` + 原型是 V1 Settings 主题交互的权威说明。

V1 Shell 视觉额外拆分 `rail/sidebar/context/content/primary-action` 语义色，保证浮动栏、侧边栏、面包屑和主内容区在 8 个主题下层级稳定。主题色只用于选中态、状态点和少量强调；危险操作固定使用 danger 色，不和普通 accent 混用。

### 4.7 悬浮层调色板独立（决策）

**决策：悬浮球与 Focus Bar（原型中的 `.fb-root` 作用域）刻意使用一套固定的浅色"玻璃"调色板，不跟随母版 `[data-theme]` 切换。**

- 理由：浮球/状态条是常驻在任意桌面/任意 App 之上的系统级悬浮层，需要在所有背景下保持稳定可读的统一质感（类比 macOS 通知中心、Spotlight 等系统浮层均不跟随当前 App 主题）。让它随母版明暗切换反而会在不同主题下忽明忽暗、与桌面环境打架。
- 实现：`.fb-root` 在原型 CSS 中硬编码自有 palette（`--surface` 浅玻璃 / `--ink` 深字 / `--blue #147af3` 等），与母版 `[data-theme]` 解耦。
- 边界：**状态语义色仍与全局对齐**——`进行中=#147af3`、分组标题强调色 `#0d9488` 等跨界面一致；独立的只是"容器底色/质感"，不是状态色板。
- 复审提示：这是有意为之，不是"浮球没适配主题"的 bug；如未来要改为跟随母版，需作为独立设计变更评估，勿当回归顺手改。

### 4.8 任务两轴状态色板（主题无关固定色）

Task 有两条独立的状态轴，色板均为固定 hex、主题无关（避免默认白主题下语义色漂移）。权威枚举/转换见 [PRD §状态](PRD.md) 与 [04-studio §2.4/§3.2](fp-ui/04-studio.md)。

## 5. 交互原则

### 5.1 非侵入

悬浮球常驻但不抢焦点（nonactivating NSPanel），面板 hover 弹出 / 离开收起，钉住才持久。

### 5.2 三层递进

悬浮球（零信息，纯入口）→ 快捷面板（紧凑列表，操作为主）→ 主看板（完整配置），信息密度逐层递增。

### 5.3 不打断工作流

弹窗失焦自动关闭（`didResignActiveNotification` → `abortModal`），PendingAction 保留上下文，回来后一键继续。

### 5.4 一致性

同一主题下 AppKit + SwiftUI 颜色统一，通过 ThemeColors ns*/sw* 双套桥接。

### 5.5 V1 主面板原型壳层

V1 新界面原型统一使用 `docs/fp-ui/00-layout-prototype.html` 作为壳层母版。活动栏、侧边栏、工作区切换、基础顶部栏由母版统一维护，页面专项原型只保留复杂流程和深水状态参考，不再重复实现全局壳层。

原型视觉遵循现有深色主题语义色：主背景、侧边栏背景、accent、三级文字、分隔线和行高亮。原型可使用 HTML/CSS 表达精确布局，但不得反向定义产品需求、数据模型或业务规则。

### 5.6 团队管理型页面

AICrew 等团队管理型页面必须让”成员”成为第一层对象，侧边栏用于成员定位，主工作区用于展示成员能力、授权状态、常驻职责与运行状态。不得把团队能力压缩成单个表单。

Runtime 是成员背后的独立执行对象，必须以绑定关系、健康状态、Owner、可见性、心跳和 Provider 被解释清楚，不得只展示为一个不可诊断的字符串。MCP、Skill、云端执行等底层概念作为成员能力属性出现，并通过状态、权限和调用边界解释可用性。Env secret、MCP JSON 等敏感配置默认展示 redacted 或 key count，只有用户显式 Reveal 后才显示可编辑值。

团队管理型页面默认不得暴露超过三层一级导航。除预置代码工程师外，其他角色默认作为模板出现，避免侧边栏看起来像已经配置完成的真实成员。

Runtime 主视图定位为执行端监控页，只保留 Runtime Header、执行器表格和 Daemon 日志三个区域。Env / Args / MCP 等编辑配置不出现在 Runtime 主面板（高级配置留给 Settings）。机器列表放在 AICrew 侧边栏 Runtime 分段中。

### 5.7 记录与跳转型交互

AICrew 的最近工作、运行记录和 Runtime 日志属于“可追溯记录”，必须同时提供摘要和定位能力。列表行以状态图标、对象编号、标题、时间、耗时为主信息，右侧使用图标按钮承载“定位到 Focus”和“查看记录”，避免把操作文字塞进行内造成拥挤。

点击规则必须稳定：点击 Task 标题或编号定位到 Focus；点击整行打开运行详情；点击成功次数、成功率或 Skill 统计打开带过滤条件的记录列表。跳转前不弹确认，目标缺失时在原位置显示“原 Task 已不存在”，并保留日志可读。

运行记录详情使用抽屉或全屏详情页，不使用小弹窗承载长日志。日志事件默认只展示摘要，展开后再查看 payload；复制、筛选、排序等工具放在顶部工具栏。secret、API key、token、cookie 等敏感字段在摘要和 payload 中默认 redacted，不允许依赖前端隐藏来保护明文。

### 5.8 AICrew 页面布局与导航规范

AICrew 页面采用侧边栏 + 工作区双栏结构，侧边栏和工作区各有独立的 Tab 导航层级。

#### 侧边栏

侧边栏顶部固定标题区（标题 + 搜索框），下方使用 `seg` 分段控件切换两个 Tab：

| Tab | 标签文案 | 内容 |
|-----|---------|------|
| 智能体成员 | `智能体成员` | 团队成员列表、新建按钮、模板列表、运行概况卡 |

#### 工作区——Runtime 视图

点击侧边栏 Runtime Tab 中的某个节点后，工作区切换为**单页执行端监控**（无顶部 Tab），顶部分段控件清空。daemon 进程负责自动扫描本机 CLI、读取环境配置、维护健康度和日志，UI 只展示状态和操作控制。

**面包屑**在 Runtime 视图下更新为 `AICrew / Runtime / {节点名}`。

**Runtime Header**（CSS class `crew-detail-header`）：节点名 + 副行摘要（`5 个执行器 · 5 个在线 · 全部空闲 · daemon v0.3.11 · 最后心跳 12s 前`）+ 在线状态 pill + 操作按钮（View logs / Restart / Stop）。Header 展示 daemon 版本，不展示单个执行器 CLI 版本（各执行器版本可能不同，CLI 版本可通过 tooltip 或日志详情查看）。

**操作按钮规则**：
- View logs：平滑滚动到底部 Daemon 日志区
- Restart：触发 daemon 重启，状态 pill 显示"重启中…"，完成后恢复
- Stop：**危险操作**，弹出确认对话框；确认后 daemon 停止，状态 pill 变为"已停止"，隐藏 `Restart` 仅保留 `Start`；恢复 online 后重新显示 `Restart` / `Stop`

**执行器表格**：`runtime-table` 表格，4 列精简——执行器（图标 + 名称）/ 健康度（pill）/ 智能体 / 工作负载。

**Daemon 日志区**：内联展示，用 `line-soft` 分隔线和"Daemon 日志"小标题标识。`刷新` 按钮手动刷新。

**远程节点未连接空状态**：整页只显示一个空状态面板（`crew-runtime-empty`），包含：图标（48px、50% 透明度）→ 节点名（h2）→ 状态 pill（`未连接`）→ 说明文案（13px、dim 色、最大宽度 360px、行高 1.6）→ 两个操作按钮（`配置连接` primary + `安装指南` 普通）。通过 `setRuntimeHostAvailability(isLocal)` 函数切换详情面板和空状态面板。

#### 通用视觉规则

- **状态点色值**：绿色 = 在线/空闲（默认），黄色 = 执行中/busy（`.busy`），红色 = 不可用/离线（`.offline`）。圆点 8px、内联到行左侧
- **pill 状态色**：`green`=在线/已连接/成功，`amber`=待授权/警告，`blue`=Skill 标签，`red`=失败/停止
- **secret-row 布局**：key 输入框 + `type="password"` 遮蔽值 + Reveal 按钮，横排排列。Reveal 仅切换当前行的显示/隐藏
- **子 Tab 分段控件**与顶部主分段控件使用相同的 `.seg` 组件，但通过 `justify-content: flex-start` 左对齐，`margin-bottom: 12px` 与下方内容分隔
- **Stop 按钮**使用 `--red` 色文字，与其他操作按钮在同一行但视觉区分危险操作

### 5.9 可编辑 chip（就地字段修改）

任务详情卡片等"信息条"场景，可枚举字段的修改统一收敛为**一种交互**——点 chip 弹同款下拉，不另设状态徽章下拉、底部操作条等多入口（符合「克制/干净」）。规范：

- **可编辑性可见**：可改字段渲染为 chip 按钮，尾部带 `▾` 小箭头、hover 高亮；只读字段（项目/归属/截止等）渲染为普通 chip，不挂 `▾`、不可点。
- **下拉定位**：单一共享下拉浮层，定位到被点 chip 正下方（锚点容器 `position: relative` + 浮层 `position: absolute`，按 `offsetLeft/offsetTop` 对齐，右侧用 `min(left, 容器宽 − 菜单宽 − 8)` 防溢出）。
- **当前值标记**：下拉项左侧 ✓ 标记当前值，**以 key 全等判断**，不依赖展示串比对。
- **数据用 key 不用展示串**：每个可枚举字段以 `{ key → label }` 模型驱动，存储 key（如 `priorityKey: "p0"`、`execmodeKey: "auto"`），展示时映射 label；新增字段只在模型加一项。状态词汇遵循看板 6 态（待规划/待办/进行中/审核中/已完成/已阻塞）。
- **关闭方式**：点同一 chip 收起、点容器外空白收起、**Esc 收起**（Esc 优先收下拉，再关卡片）。
- **即时反馈**：选定后就地刷新依赖该字段的所有区域 + 轻量 toast 提示。

## 8. 修改时必须遵守的规则

1. **配色**：新增 UI 元素必须使用 ThemeColors 取色，禁止硬编码颜色值，确保 8 主题视觉一致
2. **动画**：参照 `Constants.Design.Anim`（micro/fast/normal），节奏统一，禁止花哨过渡
3. **间距**：参照 `Constants.Design.Spacing`（xs/sm/md/lg/xl），避免随意数值
4. **图标一致**：悬浮球视觉修改必须同步 `FloatingBallView.createBrandLogoImage` 和 `scripts/gen-icon.swift`
5. **弹窗规范**：新增弹窗默认添加失焦自动关闭（`didResignActiveNotification` → `abortModal`）
6. **新建智能体弹窗**（个人版）：头像占位 + 名称（必填，空值红框拦截）+ 描述（255 字计数）+ 运行时/模型双列 select + 指令 textarea（默认收起）。个人版不展示可见性栏（Owner/可见性归后续团队版）。创建后侧边栏插入草稿成员行
7. **Studio Workspace 上下文**：当 Workspace 已由项目视图或 Session 右面板确定时，创建弹窗应灰色只读展示当前 Workspace，而不是隐藏配置区或允许改选
8. **Studio 任务视图切换**：侧边栏顶层命名为 `任务视图 / 项目视图`，不要把 `看板` 用作顶层视角名称；主区域只保留一个视图下拉，默认 `▦ 看板`。下拉菜单上方为灰色分组标题 `执行视图`，包含 `▦ 看板 / 田 泳道 / ☰ 列表`；下方为灰色分组标题 `规划视图`，包含 `▱ 时间轴`。符号使用 dim/muted 文本色，不使用彩色 emoji
9. **Studio 顶部筛选**：顶部保留 `目标` 低调下拉控件；旧 `来源` 改为组合 `筛选` 菜单，一级栏目为 `项目 / Agent / 优先级 / 标签 / 负责人 / 创建者`，二级选项支持多选，一级栏目显示选中数量。时间范围和粒度统一由左侧 Scope 控制，不在顶部重复出现。目标/组合筛选选中后同步刷新时间轴、看板、列表和泳道，激活态仅用 elevated 背景和文本加重表达
10. **Studio 时间轴分组控件**：`▱ 按目标 / ▤ 按项目` 是时间轴视图的**内嵌头部控件**（位于时间轴内容上方，不再挂在全局工具栏），仅时间轴模式可见；统一形态为 `分组方式` 文字标签（普通 span，颜色可控）+ 无 optgroup 的扁平下拉；按目标展示目标树甘特，按项目以 Workspace/项目为行轴展示同一计划范围内任务
11. **Studio 泳道分组控件**：`▤ Workspace / ◉ 执行 Agent / ↳ 父级任务 / ◎ 负责人` 是泳道视图的**内嵌头部控件**（位于泳道网格上方，不再挂在全局工具栏），仅泳道模式可见；与时间轴同形态（`分组方式` 标签 + 扁平下拉）。全局工具栏只保留视图下拉 / 目标 / 筛选，看板和列表视图无分组控件，切换视图时工具栏布局保持稳定。分组控件只改变泳道行轴，每个分组行提供轻量圆形折叠按钮，折叠态不渲染任务卡片
12. **Studio 拖拽与列表批量操作**：泳道和看板的 drop target 使用轻量描边和 `accent-bg` 高亮；泳道卡片跨 Workspace、Agent、父级任务、负责人或状态拖动后必须刷新时间轴、看板、列表、泳道和项目任务投影；看板卡片拖到其他状态列时只更新任务状态。列表模式行首使用复选框，全选/批量操作栏显示已选数量，并支持批量修改状态、优先级、负责人和删除
13. **Studio 看板卡片 Workspace 行**：跨项目看板卡片底部最后一行固定显示 Workspace 名称，使用 10px dim/muted 信息层级，避免和优先级、执行方式、Agent 状态争抢主信息
14. **Studio 对话历史折叠**：项目视图 Session 历史默认显示最近 5 条，超过部分使用灰色小字 `展开显示 / 折叠显示` 切换；不要使用醒目按钮或卡片承载该开关
15. **全局快捷对话助手**：右下角入口始终可见，默认只显示圆形聊天符号；展开面板宽度不超过 `min(520px, calc(100vw - 32px))`，高度不超过 `min(680px, calc(100vh - 48px))`。历史列表按 Workspace 分组并在组标题/行首显示 Workspace 项目符号；标题区下拉只选择已有历史，不提供新建入口；`+` 新建时进入 `新对话` 草稿态，首发后再自动提取标题；展开入口保留原展开箭头
16. **克制原则**：不加装饰性阴影、不加多余分割线、不加无意义动画，每个元素都应能回答"为什么需要它"
